1 · 起因：v5 想改 v4.2 的什么

v4.2 是"三个 LLM 角色接力（planner → executor → verifier）+ 一堆写死的 DFT 规则"， executor 拿到的工具是裸的 bash：它自己 cat 几兆的 .out、自己 grep 路径、 自己判断收敛。这套能跑，但有几个长期痛点：

  • 上下文被原始输出撑爆——cat scf.out 一次几千行进 context，模型容易看花眼、产生幻觉。
  • SCF 不收敛时原地打转——executor 在同一段上下文里反复重试，越试越乱，没有"换个脑子重新诊断"。
  • 每道题从零开始——上一道 Si 用了什么 ecut / k-mesh，这道题完全不知道。
  • 卡死没人管——模型陷入"再试一次→还是不行→再试一次"时，没有断路器把它拽出来。

v5 的设计回应（灵感来自 SWE-agent 的 ACI、Voyager 的技能库、Magentic-One 的 progress ledger）：

  1. 结构化工具层：7 个带类型的 MCP 工具替代裸 bash；read_qe_output 只给"开窗"视图，永不整段 dump。
  2. SCF 收敛子 agent（convergence_fixer）：SCF 挂了就甩给一个干净上下文的子 agent，按处方表诊断、最多试 3 次。
  3. 卡死检测 + ledger：每次工具调用后跑 7 个信号（S1–S7），命中就往上下文里塞一条 [ledger] 提醒。
  4. 经验库（Voyager 式）：把过往成功 run 的参数检索出来，拼到 prompt 前面。
  5. 程序级 verifier：verify_run 在提交前做物理 sanity，solve.py 还会复核一道硬门。
诚实的副标题：上面 5 条里，真正完整接线、跑得到的只有 1、2（部分）、3、5； 经验库（4）原先既有 params={} 的 bug、又会把空间群答案回灌进 prompt（已于 2026-05-27 审计修复，见顶部绿框）；而那张漂亮的 SCF 程序化处方表其实没接上线（详见 §3.2）。本文不替这些遮丑——恰恰相反， "哪些没接线、该不该接"正是想请你帮判断的事。
v4.2 像一个有总工程师（planner）盯着、按 SOP 手册（5 份材料 playbook + 验收清单）干活的施工队； v5 像给了一个老师傅一套更趁手的工具箱、出事能叫专家会诊，但暂时撤掉了总工程师、SOP 手册也只剩薄薄一页。 工具更好用了，但写死的工艺规范变少了。

3 · v5 里真正"代 agent 做 DFT 决策"的地方

这是本文的物理核心。v5 写进代码、约束 QE 物理的就这三块：(§3.1) 输入阈值 + k-mesh 纪律、 (§3.2) SCF 处方表、(§3.3) verify_run 验收门。我把每块的接线状态都标出来。

3.1 · 已接线的硬纪律：阈值收紧 + hex/trig k-mesh

write_qe_input 生成输入后立刻过 qe_input_lint，对 vc-relax 把三个收敛阈值 按 min(模型给的值, 默认严值) 收紧，模型只能更严不能更松：

forc_conv_thr1.0×10⁻⁴ Ry/Bohr&CONTROL
etot_conv_thr1.0×10⁻⁵ Ry&CONTROL

这条和 v4.2 一致，是从 v3 起就吃过亏总结出来的硬纪律。

六方 / 三方晶系 k-mesh ÷3 纪律 wired

写在 executor 主 prompt 里、并由 qe_input_lint 出 warning：六方/三方晶系（ibrav=4/5 或晶格角≈120°/60°）的 SCF k-mesh 的 N_x、N_y 必须能被 3 整除，才能采到 K=(⅓,⅓,0) 和 H=(⅓,⅓,½) 角点——而 VBM/CBM 常落在那里（BN_hex、石墨烯衍生物、α-石英、hex GaN/AlN…）。

  • 推荐 hex SCF：[12,12,N_z] 或 [18,18,N_z]；不要 [8,8,_] / [10,10,_]。
  • 三方：[12,12,12] 起步，各分量 ÷3。
  • 选了非 ÷3 → SCF Fermi 偏低、评测器把 VBM/CBM 误判 ~0.1 eV → vbm_k_label=OFF_PATH、扣 0.2/1.0。
这是 v5 比较亮眼的一条真·已接线的 DFT 纪律，来自 2026-05 对 hex 题的复盘（见 git 提交 T1 k-mesh formula + T2 seekpath primitive cell）。请你重点看 §6 里它对不对。

概念

为什么用"干净上下文"的子 agent

v4.2 是 executor 在同一段上下文里原地重试 SCF——同样的推理、同样的盲点，越试越糊。 v5 的 convergence_fixer 是个 fresh-context 的子 agent：看不到父 agent 那些失败尝试，被逼着重新诊断；只给它 4 个工具（读/写/跑/验，没有 bash、没有 submit）；硬性最多 3 次。 这套调度结构是已接线的 wired——只是它脑子里那张表 是 markdown 而非程序化表（见上方真相框）。

6 · 想请你审的 4 类问题

§6.1 · 这 6 条 SCF 症状→处方规则，物理上对不对、全不全？该不该接线？

  • 逐条看 §3.2 的表：症状判据（正则匹配的那串 pw.x 报错）和处方是否合理？比如 "S matrix not positive definite → 强制 MV smearing + degauss 0.02"——这个因果对吗？ 有没有该是"先查是不是真金属、别盲目 smearing 把绝缘体也抹了"？
  • 升级链对吗？如 mixing_beta 0.7→0.3→local-TF、Davidson→CG 的顺序与取值。
  • 漏了哪些常见 SCF 失败模式（如 starting magnetization 不当、ESM/带电体系、强关联需 +U）？
  • 关键决策：既然程序化表（确定性、可在花一个 LLM 轮次前就自动重跑）物理上更可信， 值不值得把它接线、取代当前让 LLM 读 markdown 的做法？还是说"让 LLM 灵活挑"更稳妥？

§6.2 · verify_run 的 T1/T2 物理门，阈值定得对吗？

  • T1：force 1×10⁻⁴ Ry/Bohr、stress 0.05 kbar、cubic 角度漂移 0.5°、晶格漂移 5%—— 这些当"通过/打回"的判据是否合适？对低对称（单斜/三斜）体系 0.5°/1.0° 的角度容差够不够？
  • T2：|E_F|<50 eV、带隙 [0,12] eV、DOS 覆盖 E_F±2 eV、nbnd≥4—— 这些"plausibility"门是否太宽/太窄？比如宽带隙绝缘体（>12 eV，如某些氟化物）会不会被误杀？
  • 关键：硬门 initial=None 跳过空间群检查（§3.3 漏洞框）， 你认为应当让硬门强制带上 initial、把空间群守住吗？（这正是 0.25 分所在）

§6.3 · v5 该不该把 v4.2 的物理层整体搬过来？

  • 有没有哪条 v4.2 的规则你其实觉得是过度拟合、v5 不抄反而更干净？

§6.4 · 经验库"按过往相似 run 直接套参数"，物理上站得住吗？

  • 修复后的行为：检索过往成功 run 的 ecut / kpts / smearing / degauss 方法论拼进 prompt 当起点， 且已剥离一切产出量（空间群）、排除当前同题。现在一道 MgSiO₃ 任务拿到的是 AlAs/CaO/Au 等 别的材料的方法论参数（实测）。
