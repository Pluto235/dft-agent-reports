1 · 起因：v5 在 deepseek 上为什么会超时

v5 的底盘很漂亮，但在一类题上反复 0 分：M2 的 20–32 原子 vc-relax（CsPbI₃、WO₃ 这种）。 根因不是物理错，而是参数过度收敛 + 墙钟用尽：

  • 模型（这里是 deepseek-v4-pro，走 legacy 路径）自己挑了 ecutwfc=80, ecutrho=800， 又配了各向同性的 K_POINTS 8×8×8 = 512 个 k 点；
  • 20–32 原子的盒子在这个参数下，单道 vc-relax 的墙钟超过 harness 的 ~4 h 上限 → 进程被回收 → 该题直接计 0。
  • 而这些题是按 1% 相对晶格容差打分的，不是按绝对总能 —— 这个容差在 ecut=50 就已经满足 （任务 JSON 自带的 ecut 扫描可证），ecut=50 + 合理 k 网格 ~1.5–3 h 就能跑完。
核心概念

converge ENOUGH, not maximally

对这些 vc-relax 题，"更高的 ecut / 更密的 k" 不会换来更高的分，只会换来更长的墙钟——而墙钟一旦用尽，分数从可能的 1.0 直接掉到 0。过度收敛在这里是纯粹的负收益。v5.1 的全部改动都围绕这一句话。

v5 像个追求极致精度、却没带表的工程师——他能把活干得很漂亮，但常常在交付截止前还在"再多收敛一位小数"， 结果什么都没交上去。v5.1 没换人，只是塞给他一张便条（preflight + 三条 lesson）："这活 50 Ry 就够， 长轴别过采样，留好时间，跑完一次就交"。便条不强制，但他照着做了。

3 · v5.1 改了什么（核心）

相对 v5，v5.1 只做了三件事：(§3.1) 一个共享 preflight 把材料知识喂进 prompt、(§3.2) 三条写死的 系统提示词 lesson、(§3.3) 移植 v4.2 知识层时顺手补的两个基准泄漏。底盘没动。

3.1 · preflight 知识层（确定性，喂两条路径）wired

入口是 agent/preflight.py:run_preflight(prompt_text, workdir)，在 solve.py:amain 里 紧挨着经验检索之后调用（solve.py:395–406）。它串三步、产出一段 advice 文本块，由 assemble_user_prompt 拼到 prompt 最前面（顺序：preflight ▸ experience ▸ 原始 prompt 殿后）：

步模块（v4.2 移植）干什么对 CsPbI₃ 的输出
1 structure_features.extract 从 prompt 解析晶系 / 化学式 / 元素 / 材料家族 / 原子数（纯解析，无 LLM） orthorhombic, CsPbI₃, [Cs,Pb,I], halide, nat=20
2 skills.select_and_render（5 份 advisor） 按 YAML frontmatter 的 trigger（material_family / crystal_system 谓词， any_of/all_of）匹配材料家族 playbook 无 skill 命中（halide 不在 5 份里）→ 这一步空；WO₃ 则命中 low-symmetry-oxide
3 config_advisor（仅确定性分类器） 按元素集判金属/绝缘体 → occupations / smearing / ecutrho_factor / nspin 建议； 落盘 .config_advice.json 含电负性 I → 绝缘体 → occupations=fixed, ecutrho_factor=10, nspin=1
概念

为什么是"确定性"且"永不崩"

preflight 全程没有 LLM 调用——纯 Python 规则，所以又快又可复现。它在两条路径前都跑，等于给任何模型 一个统一的物理起点。run_preflight 的每一步都包在 try/except 里：任何子步出错只降级为"部分/空"块， 绝不让 preflight 拖垮 solver。注意 config_advisor 的 LLM 交叉校验 (config_advisor.run()) 没有启用——v5.1 只走确定性分支（启用它属于 Phase 2，且当前直接调会 AttributeError，见 §7 P2-2）。wired（确定性）

3.2 · 三条写死的 lesson wired

在 agent/system_prompt.py 顶部加了一段以 ## V5.1-LESSON 为标记的固定文本，拼进系统提示词最前。 原文（核对自 system_prompt.py:75–93）：

## V5.1-LESSON — converge ENOUGH, not maximally (wall budget is real)

These vc-relax tasks are graded on a 1% relative lattice tolerance, NOT on
absolute total energy. Over-converging wastes wall time and TIMES OUT.

- ecutwfc: start at 50 Ry (ecutrho = 10×ecutwfc for USPP/GBRV). 50 Ry
  lands inside the 1% lattice tolerance for typical cells; pushing to 80 Ry
  with a dense mesh can time out 20-32 atom cells at the ~4h harness wall.
- k-mesh: use an ANISOTROPIC, density-based grid, not an isotropic N×N×N.
  For a cell with very different a/b/c (e.g. a≈4.8, c≈18 Å) a uniform 8×8×8
  over-samples the long axis and wastes hours. Scale each direction inversely
  to its lattice constant ... ~40-50 k-points per inverse-angstrom-equivalent.
- Respect your wall budget. ... If you choose parameters that won't
  finish, you score ZERO. Do one well-parameterized vc-relax; do not re-run
  an already-converged cell to "double check".

三条分别对应 §1 根因的三个面：(1) ecut 起点 50 Ry、(2) 各向异性密度型 k 网格（按各轴 实空间晶格长度反比取点，长轴稀、短轴密）、(3) 守住墙钟预算（不收敛完就交=0；已收敛别"再跑一遍确认"）。 这是纯 prompt 纪律——它劝模型，但（在 legacy 路径上）不强制。§6 会看到 deepseek 确实照做了。

3.3 · 移植 v4.2 知识层顺手抓出的两个基准泄漏

把 v4.2 的知识层搬进来时，做了一遍 benchmark-integrity 审计，发现有两处会把 bench 答案回灌进 模型——一处是底盘里旧有的、一处是随知识层一起搬进来的。两处都已修。理解 v5.1 知识层"现在到底装了什么"， 必须知道这两处，因为它们决定了 advisor 和经验库的当前形态。

① 经验库回灌空间群答案（底盘旧疾）已修 wired

提交 182f431。经验库每条 row 存了 space_group_number——而空间群正是 bench 给 T1 打分的 产出量（占 0.25 分）；检索 retrieve_for_task 按化学式匹配、不排除当前 task_id、专挑 score≥0.7 的成功行 → 重复材料的同题正确空间群被原样写进 "Past similar runs" 块 （已实测：MgSiO₃ 曾被喂回 sg=62, score=1.00）。

修法 4 处：query(exclude_task_id=...) 排除当前题；format_experience_block 不再打印任何产出量 （只留 ecut/k-mesh/smearing/walltime 方法论）；store.add() 写入即 scrub_answers()， space_group_number 永不落盘、也不再当 embedding key；清库脚本 scrub_experience_answers.py 把数据 130 → 13 行（删 117 行无参数垃圾 + 抹掉空间群）。

② 知识层里的"答案查表"（随 v4.2 移植进来）已修 wired

提交 21818f0。移植 v4.2 的 config_advisor + _metals.py 时发现里头藏着按化学式 查 bench 答案的表，三个失败模式：

  • _metals.py 有三张 frozenset[str]（PBE_NEAR_METAL_FORMULAS / MAGNETIC_TM_OXIDES / METALLIC_TM_OXIDES），内容是一道题一道题加进去的—— 交叉核对 bench 题库，每个化学式都对应一个 bench task ID，这些表实质是"这道题的标准协议是 X"的答案查表， 而非通用化学规则。
  • config_advisor.render_config_advice_block 往系统提示词里直接注入 "Bench GT confirms gap_type=metal, band_gap_ev=0" 这种把方法论变成答案归因的句子， 还夹带 "CoSb₃ 2026-05-25 塌到 0.16 的根因…修后 ~0.95" 这类历史叙事。
  • system_prompt.py 有多处 "match bench GT canonical" / "verified on AlAs 2026-05-21" 之类 点名 bench task ID 当物理配方来源的归因行。

修法：把化学式白名单换成元素集化学规则——_metals.py 现在是 METAL_ELEMENTS / STRONGLY_ELECTRONEGATIVE 两个元素集 + 启发式函数 is_likely_magnetic_3d_oxide() / is_likely_narrow_gap_heavy_p()（按"开壳 3d 过渡金属 + O"、 "重 p 区窄隙阴离子"这类元素特征判断，不点名任何化学式）；advisor 渲染的措辞改成"化学上倾向于…"， 绝不说某个评分系统'确认'了什么。

要点

v5.1 知识层"现在装的是什么"

元素集化学规则 + 已脱敏的方法论经验，不是化学式→答案表。这是理解 §3.1 那个 config_advisor 输出的前提：它给的是"含电负性 I → 倾向绝缘体"这种从元素推的结论，而不是 "CsPbI₃ 这道题答案是绝缘体"。两个泄漏修完后 pytest 仍绿（参 v5 文档同款审计）。

4 · preflight 怎么工作：CsPbI₃ 走一遍

用真实产物把 §3.1 的三步串起来看。下面是 CsPbI₃ 那道题 workdir 里 preflight 落盘的 .config_advice.json（确定性分类器的输出，原样）：

{
  "formula": "CsPbI3",  "elements": ["Cs", "Pb", "I"],
  "structure_name": "orthorhombic perovskite",
  "pseudo_family": "GBRV_USPP_PBE_v1.5",  "is_uspp": true,
  "is_metal": false,          "occupations": "fixed",
  "smearing": null,           "degauss": null,
  "ecutrho_factor": 10.0,     "recommend_nspin": 1,
  "magnetic_tm_oxide": false, "metallic_tm_oxide": false,
  "rationale": "contains electronegative atom(s) ['I'] -> insulator/semiconductor
                (ionic or covalent bonding)",
  "source": "deterministic"
}

关键在 rationale：判据是"元素里有电负性 I"，不是"CsPbI₃ 这道题"。由此推出 occupations=fixed（绝缘体不开 smearing）、ecutrho_factor=10（USPP 的 ecutrho=10×ecutwfc）、 nspin=1（非磁）。这段连同三条 lesson 一起进了 prompt。

skill 这一步对 CsPbI₃ 是空的：5 份 advisor（alkali-metal / complex-silicate / low-symmetry-oxide / polar-III-V / layered-vdw）没有 halide 钙钛矿这一类，所以 select_and_render 返回空——CsPbI₃ 的物理建议 全靠上面的 config 块。WO₃ 则不同：它是单斜氧化物，命中 low-symmetry-oxide-advisor （trigger：material_family: oxide + crystal_system: monoclinic），那份 advisor 专门讲 "各向异性 k 网格、按实空间晶格长度反比取点"——正好压中 §1 的根因。

6 · 跑一题：CsPbI₃ 从超时到 1.0 + 成绩单

顺一遍 CsPbI₃ 在 deepseek-v4-pro / legacy 路径上的真实轨迹（产物取自 runs_local/2026-05-28T01-35-17Z_M2_T1_CsPbI3_gamma_vc_relax/workdir）：

solve.py amain()
├─ parse_required_keys(prompt)       → ["vc_relax_out"]
├─ run_preflight(prompt, workdir)    → 落盘 .config_advice.json
│     elements=[Cs,Pb,I] → 含电负性 I → 绝缘体 → occupations=fixed, ecutrho×10, nspin=1
│     skills: 无命中（halide 不在 5 份 advisor 里）
├─ retrieve_for_task(...)            → 别的材料的方法论参数（排除本题；不含空间群）
├─ assemble_user_prompt              → [preflight ▸ experience ▸ 原始 prompt]
│                                       系统提示词最前还有 V5.1-LESSON（50 Ry / 各向异性 k / 墙钟）
└─ legacy run_loop（deepseek）
   ├─ 写 vc_relax.in：ecutwfc=50, ecutrho=500, occupations='fixed'   ← 照 lesson + config
   │                  K_POINTS automatic  9 4 3                       ← 各向异性！
   │                  （盒子 a=4.85, b=10.65, c=18.03 Å → 短轴密、长轴稀）
   ├─ run pw.x（vc-relax, nat=20）→ BFGS 每步 6–8 次电子迭代收敛
   ├─ 终态：press≈-0.02 kbar（<0.05 阈值），所有角 90.00°，Pnma 对称保持
   │         a=4.887, b=10.876, c=18.193 Å，墙钟 ~79 min
   └─ submit({"vc_relax_out": "vc_relax.out"})
→ result.json：total_score = 1.0

这条轨迹正是 §3 的三件事在协同：config 块给了 occupations=fixed，lesson 给了 ecutwfc=50 与"各向异性 k"，模型据此写出 9 4 3（沿 c=18 Å 的长轴只取 3 个点，沿 a=4.85 Å 短轴取 9 个）， 单道 vc-relax ~79 min 跑完、空间群保持 → 1.0。同样的题在 v5 下是 ecut=80 + 8×8×8 → 超时 0 分。
