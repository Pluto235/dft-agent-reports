1 · v5.1 是什么 + 为什么有这一版

它是什么：bench 给每道 DFT 题准备一个 task_prompt.txt（材料、初始晶胞、要交哪些输出）， 启动 solve.py 作为子进程。solve.py 驱动一个大模型，一轮一轮地调用工具把这道题做完， 最后写 result.json，由 bench 打分。模型不是裸跑 shell——它拿到的是带类型的结构化工具： "写 QE 输入"、"跑 pw.x"、"开窗读输出某个片段"、"解析弛豫后结构"、"物理验收"、"提交"。§2 把这套结构讲透。

为什么有 v5.1：上一版 v5 的底盘很好，但在一类题上反复 0 分——M2 的 20–32 原子 vc-relax （CsPbI₃、WO₃ 这种）。根因不是物理错，而是参数过度收敛 + 墙钟用尽：模型自己挑了 ecutwfc=80, ecutrho=800 加各向同性 8×8×8 = 512 k 点，单道 vc-relax 撞穿 harness 的 ~4 h 上限被回收 → 计 0。而这些题按 1% 相对晶格容差打分（不是绝对总能），这个容差在 ecut=50 就满足。

核心概念

converge ENOUGH, not maximally

对这些 vc-relax 题，"更高 ecut / 更密 k" 不换分，只换更长墙钟——墙钟一旦用尽，分数从可能的 1.0 直接掉到 0。 过度收敛在这里是纯负收益。v5.1 的全部新增都围绕这一句：把"够用就行"的物理纪律喂给模型。

2.1 · 入口与 prompt 装配

solve.py:amain 先解析任务：从 prompt 抽出 required keys（要交哪些输出）、推断 task_type（vc_relax 还是 bands_dos）。然后把三块文本拼成一个 user_prompt （assemble_user_prompt），顺序固定：

preflight 块（材料知识：skills + config 建议，§4）
   ▸ experience 块（过往相似成功 run 的方法论参数，已脱敏）
      ▸ 原始 task prompt（殿后，保证它的"提交说明"是最新鲜的上下文）

这个 user_prompt 两条路径都用（§3）。下面 §2.2–2.6 讲SDK 路径（现在 claude 和 deepseek 都走它）。

2.2 · executor 主循环（SDK 路径）

核心在 agent/runtime/driver.py 的 Driver。它调 claude_agent_sdk.query(prompt=user_prompt, options=...)， 拿到一个异步消息流，一条条处理。关键配置（_build_options）：

  • system prompt = agent/prompts/executor.md 渲染后的文本（build_executor_prompt 只替换 {workdir}/{pw_x_path}/{qe_backend}/{required_keys} 四个占位符）；
  • 工具 = 7 个结构化 MCP 工具（mcp_servers={"dft_qe": ...}）+ Agent（用来叫子 agent）；
  • 子 agent = {"convergence_fixer": ...}（§2.5）；
  • hooks = 每次工具调用前后各跑一个钩子（PreToolUse 写 transcript；PostToolUse 记 ledger + 查卡死，§2.4）；
  • max_turns（默认 60）、permission_mode="bypassPermissions"、cwd=workdir、 env=extra_env（给 deepseek 转发 base URL + key，§3）。

循环什么时候停：模型调 submit 工具时会置一个 ctx.submit_payload 哨兵，driver 立刻 break； 或者收到 ResultMessage；或者 max_turns 用完（交不完整）。早退时 driver 会 aclose() 那个异步生成器 （944602f 顺手修的一个 GC 报错）。

你可以把它想成给一个 DFT 老师傅配了一套规范化的仪器面板：他不能乱接线乱敲命令，只能按面板上的几个按钮 （写输入、跑、读、验、交）操作；每按一次，旁边的记录员（ledger）就记一笔，发现他原地打转就递张纸条提醒； 活干完按"提交"，质检员（verifier）再过一道关。

2.3 · 7 个结构化工具

定义在 agent/tools/structured/server.py。给模型"带类型的动作"，而不是裸 bash——这样能在写入时强制收紧阈值、 读输出时只开窗（防上下文爆炸 + 幻觉）。

工具干什么
write_qe_input按 calc=scf/nscf/bands/relax/vc-relax 生成 pw.x 输入，登记 calc_id；vc-relax 阈值自动收紧
run_pw按 calc_id 跑 pw.x；只回简短摘要（尾部 ~80 行）；命令落 bash.log 供审计；非收敛/无 JOB DONE. 标 failed（C4，§6）
read_qe_output开窗读输出：tail/errors/forces/stress/final_cell/scf_steps/bands_table/summary/grep:PATTERN，永不整段 dump
parse_structurepymatgen 取弛豫后晶格 + 空间群（T1 提交前必跑——空间群占 0.25 分）
verify_run物理 sanity，回 pass / soft_fail / hard_fail（详见 §2.6）
bash逃生舱：只在没有结构化工具时用（如 bands.x/dos.x 后处理）；仍落日志、禁后台
submit置 submit_payload 哨兵，driver 退出循环

2.4 · 卡死检测 ledger（S1–S7）wired

纯 Python、无 LLM 的断路器（agent/runtime/progress_ledger.py）。每次工具调用后， PostToolUse 钩子记一笔，并按 S1→S7 顺序查"卡住"信号，命中第一个就往模型上下文塞一条 [ledger] You appear stuck (signal=Sx...) 提醒，给 (a) 换参数 / (b) 甩收敛子 agent / (c) 交不完整 三个选项。 同一信号一个 session 只触发一次。

信号触发条件
S1近 6 轮内同一 (工具, 参数哈希) 调用 ≥ 3 次
S2近 8 轮内工具报错 ≥ 6 次
S3≥ 600 s 没有一次 verify_run pass
S4近 10 轮内同一条 bash 命令（去掉 cd）≥ 3 次
S5近 4 轮内"挫败语"正则命中 ≥ 3 次（"let me try again / keeps failing …"）
S6近 15 轮内没有一次 pw.x 成功、也没有 verify pass
S7输入 token 用了 ≥ 70%，却还没有一次成功 verify
注：S2/S3/S6/S7 依赖"识别出 verify pass / 工具报错"。在 SDK 路径上这两个判定曾经失效 （SDK 会剥掉自定义返回字段、用驼峰 isError），944602f 的 C1 修好了它们（§6）。

2.5 · SCF 收敛子 agentwired

agent/subagents/convergence.py 注册了一个叫 convergence_fixer 的子 agent。SCF 挂了，主 executor 可以叫它（Agent(name="convergence_fixer", ...)）。它是干净上下文——看不到父 agent 那些失败尝试， 被逼着重新诊断；只给 4 个工具（read_qe_output / write_qe_input / run_pw / verify_run，没有 bash、没有 submit）； 按一张"症状→处方"playbook（agent/prompts/convergence_subagent.md）选补丁，最多试 ~3 次。

2.6 · verifier 验收门 + 重放wired

模型 submit 后，solve.py:_gate_check 在最终落盘前做一道强制物理验收 （verify_vc_relax / verify_bands_dos）。任一非 advisory 项没过 = hard_fail → 把具体不满意点拼进 prompt，重放（最多 --max-verifier-replays，默认 2 次）。

T1 vc_relax 检查（非全列）T2 bands_dos 检查（非全列）
文件存在且 ≥1024 B · 末尾含 JOB DONE. · SCF 收敛 · max|F| ≤ 1e-4 Ry/Bohr · max|σ| ≤ 0.05 kbar · 空间群保持 · 晶格漂移/角度（advisory） scf/bands/dos 四文件齐全 · SCF 收敛 · |E_F| 有限且 <50 eV · bands.dat 可解析、nbnd≥4 · DOS 行数>10、窗覆盖 E_F±2 eV
这里有个历史伏笔：v5 的强制门曾用 verify_vc_relax(initial=None)，把空间群/晶格/角度检查全跳过了 ——守护 0.25 空间群分的门其实没查空间群。944602f 的 C2 修好了（从任务输入晶胞取 initial 事实再喂进门，§6）。

3 · 路由：claude 和 deepseek 现在都走 SDK 路径（本次新增）

v5.1 其实有两条执行路径：上面 §2 讲的 SDK 路径（结构化工具那套），和一条复用 v4 的 legacy 手写 tool-use loop（裸 bash）。944602f 改了分流规则，把 deepseek 从 legacy 挪到了 SDK。

分流就两行（solve.py，原文）：

def should_use_sdk_path(model, *, base_url):
    # claude 原生 → SDK；或者只要配了 Anthropic 兼容 base URL（如 deepseek 的）→ SDK
    return model.startswith("claude-") or bool(base_url)

def resolve_anthropic_base_url(model, *, cli_base_url, ...):
    # 优先级：--anthropic-base-url 旗标 > ANTHROPIC_BASE_URL 环境变量
    #        > deepseek-* 的内置默认 https://api.deepseek.com/anthropic
    ...
    if model.startswith("deepseek-"):
        return DEEPSEEK_ANTHROPIC_BASE_URL   # = "https://api.deepseek.com/anthropic"
    return None
模型base URL走哪条路
claude-*—SDK 路径
deepseek-*自动取 api.deepseek.com/anthropicSDK 路径（本次新增）
其它非 claude（无 base URL）无legacy 手写 loop（fallback）

deepseek 通过 Anthropic 兼容端点接进 SDK：API key 从 DEEPSEEK_API_KEY 取（其次 ANTHROPIC_AUTH_TOKEN/ANTHROPIC_API_KEY），连同 base URL 经 Driver.extra_env 转发给 SDK 子进程；模型 id 原样传、不重映射。意义：deepseek 现在也能用上 §2 那整套结构化工具 + 卡死检测 + 验收门，而不再是裸 bash。

⚠ 一个要诚实说清的"接线缺口"（lesson-gap）：§4.2 那三条写死的 lesson（50 Ry / 各向异性 k / 守墙钟） 目前只在 legacy 的 system_prompt.py 里，没进 SDK 的 executor.md——而且 executor.md 自己的示例还写着 ecutwfc=60 ... kpts=[8,8,8]（各向同性，正是要避免的反例）。 build_executor_prompt 只渲染 executor.md，不拼 lesson（已核对）。

后果：deepseek 既然挪到了 SDK 路径，就不再自动拿到这三条 lesson了——SDK 路径上只有 preflight 块带物理纪律（config 建议永远在；各向异性 k 的建议只对命中 skill 的材料如 WO₃ 才有，§4–5）。 所以 §8 那些 1.0 的成绩是legacy 路径跑出来的（lesson 当时生效）；deepseek-on-SDK 目前只验证了 M1_T1_Si=1.0（Si 小，本来也不需要这三条 lesson）。把三条 lesson 也接进 SDK 工具层，是 Phase 2 的头号事（§9 P2-1）。

4 · v5.1 在底盘上加的知识层

相对 v5，v5.1 在 §2 那套底盘上加了三样东西：(§4.1) 一个共享 preflight、(§4.2) 三条写死的 lesson、 (§4.3) 移植 v4.2 时顺手补的两个基准泄漏。底盘本身没动。

4.1 · preflight 知识层（确定性，喂 user_prompt）wired

入口 agent/preflight.py:run_preflight(prompt_text, workdir)，串三步、产出一段 advice 文本，由 assemble_user_prompt 拼到 prompt 最前（§2.1）。全程无 LLM，纯规则，又快又可复现； 每步包 try/except，出错只降级为空块，绝不拖垮 solver。

步模块（v4.2 移植）干什么
1structure_features.extract从 prompt 解析晶系/化学式/元素/材料家族/原子数
2skills.select_and_render（5 份 advisor）按 frontmatter trigger 匹配材料家族 playbook（如单斜氧化物→各向异性 k）
3config_advisor（仅确定性）按元素集判金属/绝缘体 → occupations/ecutrho_factor/nspin 建议；落盘 .config_advice.json

注：config_advisor 的 LLM 交叉校验（run()）没启用（直接调会 AttributeError，属 Phase 2 P2-2）。

4.2 · 三条写死的 lessonwired（仅 legacy 提示词）

在 agent/system_prompt.py 顶部、以 ## V5.1-LESSON 标记的固定文本： (1) ecutwfc 起步 50 Ry（ecutrho=10×）、(2) 各向异性密度型 k 网格（按各轴晶格长度反比取点， 长轴稀短轴密）、(3) 守住墙钟预算（不收敛完就交=0，已收敛别"再跑一遍确认"）。

这三条目前只在 legacy 提示词里，没进 SDK 的 executor.md（见 §3 的 lesson-gap 框）。 也就是说 deepseek 挪到 SDK 后，这三条暂时够不着它了——这是 v5.1 当前最该补的接线。

4.3 · 移植 v4.2 顺手抓出的两个基准泄漏已修

搬 v4.2 知识层时做了一遍 benchmark-integrity 审计，发现两处会把 bench 答案回灌进模型，都已修。理解 v5.1 知识层"现在装的是什么"必须知道这两处。

① 经验库回灌空间群答案（底盘旧疾，182f431）

经验库每条 row 存了 space_group_number——正是 T1 打分的产出量（0.25 分）；检索按化学式匹配、 不排除当前 task_id、专挑 score≥0.7 → 重复材料的正确空间群被原样写进 "Past similar runs"（实测 MgSiO₃ 曾被喂 sg=62, score=1.00）。修法：排除同题 + 产出量不入 prompt/不落盘 + 写入即 scrub_answers()；清库 130→13 行。

② 知识层里的"答案查表"（随 v4.2 移植进来，21818f0）

_metals.py 原有三张按化学式查的 frozenset（每个化学式都对应一个 bench task ID，实为答案查表）； config_advisor 还往提示词注入 "Bench GT confirms…"。修法：换成元素集化学规则 （METAL_ELEMENTS/STRONGLY_ELECTRONEGATIVE + is_likely_magnetic_3d_oxide() 等启发式）， 措辞改成"化学上倾向于…"，绝不说某评分系统"确认"了什么。

要点

v5.1 知识层"现在装的是什么"

元素集化学规则 + 已脱敏的方法论经验，不是化学式→答案表。§5 的 CsPbI₃ 例子里你会看到 它给的是"含电负性 I → 倾向绝缘体"这种从元素推的结论。

5 · preflight 走一遍：CsPbI₃

用真实产物把 §4.1 串起来。下面是 CsPbI₃ 那道题 workdir 里 preflight 落盘的 .config_advice.json（原样）：

{
  "formula": "CsPbI3",  "elements": ["Cs", "Pb", "I"],
  "structure_name": "orthorhombic perovskite",
  "is_uspp": true,       "is_metal": false,
  "occupations": "fixed", "ecutrho_factor": 10.0, "recommend_nspin": 1,
  "rationale": "contains electronegative atom(s) ['I'] -> insulator/semiconductor",
  "source": "deterministic"
}

判据是"元素里有电负性 I"，不是"CsPbI₃ 这道题"。由此推出 occupations=fixed（绝缘体不开 smearing）、 ecutrho=10×ecutwfc（USPP）、nspin=1（非磁）。这段进了 user_prompt。

skill 这步对 CsPbI₃ 是空的：5 份 advisor（alkali-metal / complex-silicate / low-symmetry-oxide / polar-III-V / layered-vdw）没有 halide 钙钛矿这一类。WO₃ 则命中 low-symmetry-oxide（trigger：氧化物 + 单斜），那份 advisor 专讲"各向异性 k、按晶格长度反比取点" ——正好压中 §1 的根因。这也是 §3 lesson-gap 下 SDK 路径上 WO₃ 仍有各向异性 k 提示、而 CsPbI₃ 没有的原因。

意义

v5.1 把 v5 评审里两处"物理智慧被架空"的地方接回了线

v5 评审当时的两个核心吐槽——守 0.25 空间群分的硬门却没查空间群（initial=None）、 SCF 处方串错 namelist——在 v5.1 里分别由 C2、C3 修好。加上 C1（SDK 路径上卡死信号真的会响了）和 C4（非收敛 run 不再冒充成功），v5.1 的"验收 + 断路"这层比 v5 实在了一截。
