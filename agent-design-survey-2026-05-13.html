DFT Agent 架构调研：跨域 SOTA 模式提炼

背景：当前 solver 是"裸 ReAct"（单 agent + bash + submit），团队评估认为远低于 2026 年标准。 本报告并行调研 4 个领域（DFT 专域 / 数学&形式化证明 / 通用 agent 架构 / 代码 agent 工程经验）的 SOTA 系统， 提炼可借鉴模式并给出针对当前 solver 的改造方案。
调研日期：2026-05-13 ・ 覆盖系统数：30+ ・ 调研方式：4 路并行 web research subagent
一句话结论：现在的 solver 是 2024 年水平。SOTA DFT agent（DREAMS / QUASAR / VASPilot）都在做 结构化工具 + 收敛专家子 agent + 程序级 verifier + 经验库 + stuck detection。 这五件事按 ROI 排序就是接下来的改造路线图。

三、按 ROI 排序的 TOP 10 招

🔥 必做（高 ROI、低实现成本）

1 把 bash 拆成结构化工具集

来源：SWE-agent ACI + Anthropic tool design

这是单点 ROI 最高的改动。SWE-agent 在 SWE-bench 上从 0% 跳到 12.5% 主要靠的就是这个。

write_qe_input(path, calculation, system={ecutwfc, ibrav, ...}, electrons, k_points, ...)
  → 写入前 lint：未知字段 reject、pseudo 文件 existence、ibrav 与 CELL_PARAMETERS 冲突
read_qe_output(run_id, view: Literal["summary","scf_steps","errors","forces","tail:N","grep:..."])
  → 默认返回 30 行结构化 summary，永远不让 LLM 看完整 pw.out
run_pw(input_path, ngpus, timeout) → 返回 run_id 而非 stdout
verify_run(run_id)                 → 硬性 verifier：收敛 + 力 + 应力 + 警告关键字
parse_structure(path)              → 解析 CELL_PARAMETERS + ATOMIC_POSITIONS

保留 bash 但只作为 escape hatch，不是默认路径。

2 加 SCF Convergence Sub-Agent

来源：DREAMS / GENIUS

DREAMS 在 Sol27LC 上 lattice MAPE <1% 的关键设计：SCF 不收敛时不让主 agent 自己想，分派给一个专门 prompt 的子 agent，输入 = (input, output, error) 三件套，输出 = (mixing_beta, smearing, mixing_mode, electron_maxstep) 的具体调整。

为什么有效：主 agent 上下文里塞了一堆历史尝试已被污染；专家 sub-agent 拿干净 context 看本次失败，建议质量好得多。GENIUS 的实验显示，失败后 context reset 比堆积历史更有效——错误尝试反而误导后续生成。

3 加 stuck detection + 强制 backtrack

来源：Magentic-One progress ledger

当前 solver 大概率在同一个错误上反复打转直到 max_turns 烧完。Magentic-One 的核心是每个 inner loop tick 回答五个问题，其中之一是"团队是否在原地打转？"，>2 就跳出 inner loop 让 orchestrator 重新规划。

  • 检测：连续 3 turn 没有新文件写入 + 没有新 error 类型 → 进入 REMEDIATION_MODE
  • 强制：注入预定义 playbook（"按顺序尝试：mixing_beta↓ → smearing 加宽 → diag='cg' → ecut+20%，最多 3 次"）
  • 兜底：3 次还不行就 submit_best_effort() 强制退出，别再烧 GPU 了

4 Experience / 教训库

来源：Reflexion + ChemAgent + Voyager

已经跑了上百个 benchmark task，结果全是一次性的——巨大浪费。建一个 KV store：

# 写入：每个成功 run
key   = embedding(material_composition + calc_type + pseudo_family)
value = {ecutwfc, k_grid, mixing_beta, occupations, success: True, walltime}

# 读取：新任务开始时 top-5 检索
prior = retrieve_similar(task, k=5)
# 注入 system prompt: "类似系统的成功设参参考：ecut≈60, k-grid 8×8×8, mixing_beta=0.4"

这一招对 benchmark 场景尤其香——任务大量重复（不同金属、氧化物、半导体），prior 命中率会很高。 ChemAgent 三分记忆映射到 DFT：Planning Memory（pipeline 范式）/ Execution Memory（材料→成功 input file）/ Knowledge Memory（QE 参数陷阱 lessons）。

5 独立 verifier 工具 + submit 前强制调用

来源：CRITIC + SWE-agent linter-gated

已经有 bench audit，但只查 "JOB DONE." 这种表面指标。需要的是物理合理性的程序 verifier：

verify_dft_run(run_id, task_type) → {
  scf_converged:      bool,    # SCF iter 数 vs 阈值
  forces_below_thr:   bool,    # max force < forc_conv_thr
  stress_below_thr:   bool,    # vc-relax: stress < press_conv_thr
  symmetry_preserved: bool,    # SpacegroupAnalyzer 前后对比
  magnetic_sensible:  bool,    # 总磁矩 vs 预期
  warnings_clean:     bool,    # 无 "negative charge" 等
  cross_check_MP:     bool|None, # 可选：和 Materials Project 比 ±5%
  verdict:            "pass" | "soft_fail" | "hard_fail",
  reasoning:          str      # 给 agent 看的诊断
}

关键：不通过不让 submit，agent 拿到 verdict 必须修复或显式放弃。这是 MatClaw 强调的"silent quality failure"防御——表面收敛了但参数其实不够好的隐性问题，最容易蒙混过关。

🟡 值得做（高价值但实现复杂）

6 Sketch-then-Fill 双层 LLM

来源：DeepSeek-Prover V2 / Plan-and-Execute

让强模型（Opus）一次性写 high-level plan：

Plan for M1_T1_Si_vc_relax:
  step 1: write vc-relax input with PBE PAW pseudo, ecutwfc=60, kgrid 8x8x8   [filler: haiku]
  step 2: run pw.x, verify SCF converged                                       [filler: haiku]
  step 3: check stress/force convergence, if not met → goto step 1 with new params  [filler: haiku]
  step 4: parse final lattice constant, submit                                 [filler: haiku]

每个 step 由廉价模型（Haiku）独立填充，Plan 错只重 Plan、step 错只重 step。 长链 drift 被天然切断 + 省 80% Opus token。

7 共享事实数据库 / Canvas

来源：AlphaGeometry2 SKEST / DREAMS Canvas

同一材料的 relax / bands / phonon 任务共享一个 KV store，存"已验证事实"（结构化类型，不是文本日志）：

canvas[material_id] = {
  relaxed_lattice:     np.array(...),
  converged_ecutwfc:   80.0,
  fermi_energy:        6.23,
  magnetic_moments:    [...],
  warnings_seen:       ["pseudo unsupported in GBRV"],
}

下游任务自动读取，不再重新猜参数。Bench 里同一材料反复出现时，这一招直接省 GPU 小时。

8 Periodic check-in 主动 kill

来源：QUASAR

长 vc-relax 跑 30 分钟还没收敛？现在只能等 timeout。改成每 N 分钟读一次 intermediate output， 发现"前 50 个 SCF iter 看趋势不会收敛"就主动 kill 并重启参数。H100 这种贵 GPU 上省钱效果显著。

9 Failure-as-Signal Backtracking

来源：ToT / Lean tactic prover

当前 main loop 是 try { run } catch { upraise }。改成显式 search loop：

state = TryState(ecutwfc=50, k_grid=6, mixing_beta=0.7)
while not done and depth < MAX:
    result = run_pw(state)
    if verify_run(result).verdict == "pass":
        done = True
    else:
        tactic = pick_tactic(result.failure_reason)   # 来自 playbook
        state = state.apply(tactic)                   # 显式 state transition
        history.push((state, result, tactic))
        # 失败到 dead-end 就回退到 history 里更早的 fork

形式化 + 可审计，远好于隐式 LLM 自己改。SCF 不收敛、力残差爆掉、NaN——都当作 tactic 失败而非 fatal error，触发 backtrack。

10 Self-Consistency / Multi-Path 投票

来源：数学推理域

对最终 observable（如 bandgap）跑 2–3 条独立 setting 路径（不同 pseudo family、不同 ecut+kgrid 组合达到等价收敛精度）， 结果交叉验证。一致 → 高置信交付；分歧大 → 升级到深入诊断。Bench 评分场景这相当于免费拿到 confidence score。

四、推荐重构架构

solve.py (entry) │ ├── Orchestrator (workflow，代码而非 LLM) │ 按 task_type route → relax / bands / phonon / EOS recipe │ ├── Strategist (Opus, 1 次/任务) │ 读 task + prior_experience → 出 high-level Plan (JSON) │ 强制 second-pass review（QUASAR 招） │ ├── Executor Loop (Haiku, 主力) │ 工具集（不再有裸 bash 作为默认）： │ - write_qe_input (typed, linted) │ - read_qe_output (view-restricted, 默认 30 行 summary) │ - run_pw (returns run_id) │ - verify_run (mandatory before submit) │ - parse_structure │ - retrieve_similar (experience lookup) │ - bash (escape hatch) │ ├── Convergence Sub-Agent (Sonnet, 仅 SCF 不收敛时召唤) │ 干净 context + 失败三件套 → 出参数调整建议 │ ├── Verifier (程序，非 LLM) │ pw.out parser + 物理 sanity check + MP cross-check │ submit 前强制调用 │ ├── Stuck Detector (代码) │ 连续 N turn 无进展 → 强制 compact + replan │ 超 K 次 remediation → submit_best_effort 退出 │ └── Shared State canvas.json (材料级事实, structured) progress.json (任务级历史, structured, 跨 turn 持久化) experience/ (跨任务 vector index, append on success)
架构借鉴来源：DREAMS 的分层 + Magentic-One 的 stuck detector + Anthropic 的 just-in-time loading + SWE-agent 的 ACI + Reflexion 的 experience + DeepSeek-Prover 的 sketch-then-fill。

五、反直觉点（容易踩坑）

  1. 不要急着上 multi-agent。 Anthropic 自己说 multi-agent 用 15× token，且显式点名 coding-like 任务不适合（DFT 是 coding-like）。 单 task 内别拆；cross-task（并行跑 benchmark 不同题）可以拆，但那只是 parallelization workflow，不是真正的 multi-agent。
  2. CRITIC 用工具不用 LLM。 已经有 pw.x output 这个 ground truth 工具，不要再叠一层 LLM critic， 直接写确定性 Python verifier 就够了。叠 LLM 反而引入新幻觉。
  3. Aider 的发现：edit format 选错比模型选错损失更大。 给 LLM 改 namelist 别用 sed，用 SEARCH/REPLACE 块（model 训练数据里见过的格式），鲁棒度高一个量级。
  4. MatClaw 警告：silent quality failure 是 DFT agent 最大隐患。 表面收敛了但参数不够好。这就是为什么 verifier 必须做到物理级（symmetry / force / stress），而不是看 "JOB DONE."。
  5. GENIUS 经验：失败后 context reset 比堆积历史强。 重试不是"在错误历史上续写"，是带着"教训摘要"重启。
  6. Example 目录优于 Vector RAG（对 QE 这种 namelist 语法）。 QUASAR / MatClaw 都验证：BM25 + AST chunking 检索现成 input file 比 embedding 准。
  7. Kimina-Prover 的反直觉发现：足够强的 RL + verifier 信号下，MCTS / PRM 都不是必须。 对早期版本，先把 verifier（pw.x + linter）做扎实，再考虑加搜索。

② 加程序级 verifier

加智能层

七、给老板的一句话总结

"现在的 solver 是 2024 年水平。SOTA DFT agent（DREAMS / QUASAR / VASPilot）已经在做 结构化工具 + 收敛专家子 agent + 程序级 verifier + 经验库 + stuck detection。 我打算分两阶段重构：先把 bash 换成 5 个结构化工具 + 加 verifier + 加 stuck detector（一周可见 benchmark 涨分）， 再加 experience 库和 SCF sub-agent（两周）。"
