§1 三版本 agent 是什么？

本节给 DFT 同行（默认你看 QE 像看母语，但没动过 LLM agent）扫盲。完整设计文档见 solver-versions-explained-2026-05-17.html。

先理解一个东西：LLM agent 在这里做什么？

把它当成一个会读你给它写的 README、然后自己开 shell、自己写 QE 输入、自己 pw.x、自己抓 scf.out 里 band gap、最后把结果填到一个 result.json 里交卷的「研究生」。 它每一步的「动作」叫一个 tool call：典型 4 个工具是 bash(cmd) / write_file(path, content) / read_file(path) / submit(task_outputs={...})。 LLM 每接一次「环境观察」（前一个 bash 的 stdout/stderr）就再决定下一步动作，直到调用 submit 收尾。 这一圈反复就叫 agent loop。

v1 baseline (45 KB system prompt)
最朴素 ReAct loop
单 agent + 上面那 4 个工具。System prompt 里只写「你是 DFT 专家，做 QE 计算」这类原则性指令。 LLM 自己决定要不要先 vc-relax、用多少 ecutwfc、k-mesh 怎么选。 类比：让一个聪明但没看过本组实验记录本的新研究生独立跑题。
v4.1 (vc-relax preflight)
主 agent + 1 个 preflight 子 agent
检测到任务要做 bands/DOS（T2）时，先派一个专门做 vc-relax 的子 agent把 晶格优化到收敛、把弛豫后的 CELL_PARAMETERS 和 ATOMIC_POSITIONS 直接注入主 agent 的 system prompt。类比：给研究生先指定师兄做完结构弛豫，再让他/她做电子结构计算。
v4.2 (Planner-Executor-Verifier)
三层 subagent 编排
Planner 子 agent 把任务先拆成 SCF→bands→DOS 三步（写成有依赖的 DAG）， 然后每步派一个 Executor 子 agent 真跑、再派一个 Verifier 子 agent 检查产物（不让 Verifier 写文件，只能读）。 Verifier 不通过就让 Executor 接着改。类比：项目经理（Planner）拆任务 → 工程师（Executor）做 → QC（Verifier）验，每步独立。

§4 头条发现

发现 1 · v4 系列把 band_gap 字段从 0.58 拉到 0.86—0.91，但 dos_curve 字段三版本都跪

把 21 道 T2 题的 6 个字段分别取平均，得到下面这张「失分热力图」（详见 §6）。 band_gap_ev、dos_e_fermi 在 v4.1/4.2 上有大幅改进；但 dos_curve 字段三版本平均分都是 0.02–0.25——这是 T2 0.8 天花板的最主要原因， 21 题 T2 里几乎每道都吃 dos_curve = 0 的扣分。

发现 2 · 不存在「v4.2 比 v1 更差」的 T2 案例，但存在 v4.2 比 v4.1 严重退化的单题

按规则「v1 ≥ 0.7 且 v4_2 ≤ 0.3 在同一题」找跨版本 regression——结果是 0 个。 但有一道v4.2 vs v4.1 退化的案例：GaN T2，v4.1 拿到 0.8，v4.2 跌到 0.05（5 个字段全 0）。 深挖发现是 v4.2 的字段提取把 GaN 抽成了「金属」（band_gap_ev=0, gap_type='metal'）， 而 v4.1 抽得正确（1.74 eV direct@Γ）。Verifier 子 agent 只检查 QE 跑没跑通，没有「这是已知半导体却抽成金属」的物理常识检查。 详见 §8 案例 B。

发现 3 · 某些题三版本都跪——根因更可能在「打分规则」或「题目本身物理性」

跨版本「集体困难户」（每版至少一道得 0；或 ≥2 版本得分 < 0.3）：

  • M1_T2_Rb_bands_dos: v1=0.205, v4.1=0.05, v4.2=0.05。三版都跪。 但 v1 的 notes 明确写了 "band 5 crosses the Fermi level → METALLIC (band gap = 0 eV)"—— agent 物理上是对的（Rb 是碱金属，确实没 gap），但是评分规则把 band_gap/gap_type/vbm_k/cbm_k 四个字段都按半导体打了 0。详见 §8 案例 A。
  • M2 transition-metal / 复杂晶体（CoSb3, Ti3O5, MgSiO3-bridgmanite）： 反复出现 timeout / scheduler kill / 0.5 得分天花板。这是物理上的「重型题」。

发现 4 · v4 系列在「v1 大跪题」上提分巨大

v4.2 比 v1 在同一题上提分 ≥ 0.2 的 T2 案例：

  • M1_T2_Ge: v1=0.18 → v4.2=0.80 (+0.62)
  • M1_T2_KF: v1=0.35 → v4.2=0.76 (+0.41)
  • M1_T2_MgO: v1=0.35 → v4.2=0.80 (+0.45)
  • M1_T2_LiCl: v1=0.10 → v4.2=0.50 (+0.40)
  • M1_T2_NaCl: v1=0.35 → v4.2=0.75 (+0.40)
  • M2_T2_MgSiO3: v1=0.10 → v4.2=0.50 (+0.40)

共性是「v1 把流程跑歪了（错了 NSCF k-path / 缺 vc-relax / 错了 occupations）」的题，被 v4 的子 agent 编排修好。 案例 C（Ge）。

怎么读这张图

  • band_gap_ev：v4.1/v4.2 大幅提升（0.58 → 0.86–0.91）。v4 的 vc-relax preflight 让 SCF 跑在 正确弛豫后的几何上，gap 值更准。这是 v4 系列最大单点贡献。
  • gap_type（direct/indirect）：三版本都在 0.74 附近持平。错的题里 有相当一部分是「金属被强按半导体」（Rb / Cd），是评分规则问题，不是 agent 算错。
  • vbm_k_label：三版本都在 0.58–0.62 徘徊。这是 agent 最稳定的「弱点」。 但有一个重要补丁：rubric **自己就有 'OFF_PATH' 字符串约定**——当 GT 的 VBM 不在 high-symmetry path 上时（如 MgSiO3），GT 字段值就是字符串 'OFF_PATH'。 agent 这边 evaluator 也内置了 OFF_PATH 检测逻辑， 但显然在很多 indirect-gap material 上两边对不上（一边说 'GAMMA' 另一边说 'OFF_PATH'）。 另外金属系列 GT 用 'metal' 字符串——见案例 A Rb 重判。
  • cbm_k_label：v1=0.76, v4=0.78–0.80。CBM 比 VBM 容易抽对，奇怪但持续。
  • dos_e_fermi（Fermi 能级数值）：v1 跪（0.145），v4.1/v4.2 拉到 0.39/0.43。v4 系列大量 通过子 agent 把 SCF 收敛 + DOS NSCF 的 E_F 跑对了。但仍然不到 0.5，说明 还有相当多题的 E_F 和 reference 差异显著（tolerance 是 ±0.1 eV absolute）。
  • dos_curve：全军覆没。v1=0.248, v4.1=0.021, v4.2=0.057。 这就是 T2 0.8 天花板——dos_curve 在 rubric 里权重是 0.20， 其余 5 字段全满分时总分 = 1 − 0.20 = 0.80 ✓。 评分用 curve_l2：归一化 L2 RMSE，带 ±15 eV best-shift 扫描（独立于 dos_e_fermi 字段评 EF 准不准）， tolerance = 0.1，err = 2*tol 时归 0。换句话说，EF 对齐不是失分原因（rubric 已经替你 shift 过了）， 真正失分的是 DOS 曲线形状本身和 GT reference (.dat 文件) 形状差太多。 猜测：smearing 函数 / degauss 宽度 / k-mesh 密度跟 GT 不一致 → 峰被铺平或者尖锐化了。

§7 T2 失分共性模式

把 21 道 T2 题在三版本上的 per_field_scores 全部摊开，按字段归零 (=0) 出现频率统计：

常见「字段归零」模式

  • dos_curve = 0 · 18/21 题在 v4.2 上：dos_curve 评分对 reference 的对齐方式（绝对能量 vs E−E_F）和 binning 步长高度敏感。agent 通常按「Emin = E_F − 5 eV，Emax = E_F + 5 eV，deltaE = 0.05 eV」 跑 dos.x，但 reference 可能用不同的 binning 或对齐方式。
  • vbm_k_label = 0 · 通用：α-SiO2、MgSiO3、CaTiO3、CoSb3 等复杂晶体上三版本都错。猜测：reference 的 k-label 用了特定文献的约定（如 W vs W2 vs SUMMIT），seekpath 返回的 label 与之不符。
  • gap_type = 0 · 在金属 / 半金属上：Rb（碱金属）、Cd（半金属性）反复被罚。 agent 正确识别金属性，但 rubric 期望「direct」或「indirect」字符串。
  • band_gap_ev = 0 · v1 多发，v4 少发：v1 缺 vc-relax， SCF 跑在错的几何上，gap 严重偏离 reference（阈值通常 ±0.1–0.2 eV）。v4.1/v4.2 用 preflight 子 agent 弥补。

§8 案例深挖（5 例）

案例 A · Rb (M1_T2_Rb_bands_dos) — 不是 rubric 不识别金属，是 evaluator 没把 agent 的金属输出 readout 出来

背景：Rb 是 alkali metal（碱金属），body-centered tetragonal (I4/mmm)， band 5 (Rb 5s) 横穿 Fermi → 必然金属。

原始 rubric（重要更正）：去看了 questions/M1_T2/M1_T2_Rb_bands_dos.json，Rb 的 ground_truth 实际上是：

band_gap_ev:  {value: 0.0,      tolerance: 0.1, rule: 'absolute'}
gap_type:     {value: 'metal',   rule: 'exact'}
vbm_k_label:  {value: 'metal',   rule: 'exact'}
cbm_k_label:  {value: 'metal',   rule: 'exact'}
dos_e_fermi:  {value: 2.0011,    tolerance: 0.1, rule: 'absolute'}

所以 rubric 已经预留了 'metal' 字符串！agent 这边只要 extracted_fields 抽出 gap_type='metal' / vbm_k_label='metal' / cbm_k_label='metal' / band_gap_ev=0.0 就能拿满。

那为什么 Rb 三版本全跪？看 evaluator 实际抽到了什么：

// v1   extracted: band_gap_ev=11.066, gap_type='indirect', vbm_k='OFF_PATH', cbm_k='GAMMA', dos_e_f=1.812
// v4.1 extracted: band_gap_ev=10.988, gap_type='direct',   vbm_k='GAMMA',    cbm_k='GAMMA', dos_e_f=1.993
// v4.2 extracted: band_gap_ev=10.988, gap_type='direct',   vbm_k='GAMMA',    cbm_k='GAMMA', dos_e_f=1.982
//
// GT 期望:        band_gap_ev=0.0,    gap_type='metal',    vbm_k='metal',   cbm_k='metal', dos_e_f=2.0011

诊断（正确版本）：

  • bench 的 evaluator (bench/harness/lib/evaluator.py) 有三级 metal-detection cascade （homo_lumo → electron_count → Fermi-slice），但它在 Rb 上三级都没识别为金属。 评估器看 agent 的 bands.dat，把高能空带（11 eV 处）当成 CBM，把价带顶当成 VBM， 人为构造出一个 ~11 eV gap。这是 evaluator 端的失败。
  • 为什么 evaluator 看不出金属？可能因为：
    1. agent 用了 occupations='fixed' 而不是 'smearing'， QE 把 Rb 当绝缘体打印「highest occupied level」一行，evaluator 的 homo_lumo tier 读到这行就用 HOMO/LUMO 逻辑，绕过了金属判定。
    2. 或者 agent 用的 nbnd 偏少，bands.dat 的最高能带没包含真正穿过 EF 的带，使得费米切片 落在 band gap 内。
  • dos_e_fermi 抽到 1.8–1.99 eV vs GT 2.0011——差了 0.01–0.2 eV， v4.1/v4.2 命中 ±0.1 tolerance 拿了 1.0，v1 偏 0.19 eV 失分。这部分 agent 跑得不错。
  • v1 的 dos_curve 偶然拿到 1.0（v4.1/v4.2 拿 0）很可能是 v1 用了不同的 smearing/k-mesh 配置， DOS 曲线形状碰巧和 GT reference 更接近。这是噪声不是趋势。

核心 takeaway：Rb 失分不是 rubric 设计缺陷，是 「agent 的 QE 输出格式 × evaluator 的 metal-detection 启发式」之间的接口问题。 要修 Rb 必须修两边之一：让 agent 始终用 metal-friendly 设置（smearing + 足够 nbnd）， 或者让 evaluator 更鲁棒地识别金属。

给合作者的 7 组问题：

  1. band_gap_ev 的 ±0.1 eV absolute tolerance 公平吗？
    PBE-DFT 算 band gap 已知系统性低估 30-50%。GT 用的就是 PBE 自跑值 （例：Si=0.61 eV, GaN=1.74 eV, α-SiO2=5.54 eV, MgSiO3=5.49 eV）。 即使是 PBE-vs-PBE，不同 pseudopotential 家族（GBRV / SSSP / PseudoDojo）的 gap 也会差 0.1-0.5 eV。 你们组内 reproduce 一个文献 PBE band gap 时，能稳定到 ±0.05 / ±0.1 / ±0.2 eV 哪个量级？ 是不是该把 tolerance 改成 relative 5% 而不是 absolute 0.1 eV（对宽带材料 5.5 eV 给 0.275 eV 余量，对窄带 0.5 eV 给 0.025 eV 余量）？
  2. gap_type 在 PBE 把材料算成「假金属」时怎么办？
    最戏剧性的例子：Ge 的 GT 写的是 gap_type='metal'（band_gap_ev=0.0）。 原因是 PBE 算 Ge 把它压成金属化（真实 Ge gap 0.66 eV）。所以 GT 实际上是「正确 PBE 输出」而不是「实验真值」。 agent 这边 v1 抽到 'indirect' (0.155 eV)、v4.1 抽到 'direct' (0.009 eV)、v4.2 抽到 'metal' (0.0)—— v4.2 跟 GT 完全一致拿了 1.0。但这个评分逻辑奖励的是「忠实再现 PBE 算法 artifact」，不是「物理正确」。 你们觉得这套设计 OK 吗？还是应该 GT 用 hybrid functional / GW 算的真值， 把 「PBE artifact」当成 agent 的另一种失分？
  3. vbm_k_label = 'OFF_PATH' 这个约定 agent 不知道（rubric 故意不暴露给 agent 防作弊）。

  4. 金属的 vbm_k_label / cbm_k_label = 'metal' 字符串——agent 端 extracted_fields 没出这个串就归 0。
    看 Rb 三版本：agent 跑出来是金属（band 5 横穿 EF），但 evaluator 在 agent 的 bands.dat 上没识别出金属， 反而抽出 11 eV 假 gap。evaluator 的三级 metal-detection cascade （homo_lumo line → electron_count → Fermi-slice）在 Rb 上全失效。你们做 Rb / Cs / Na 这类碱金属 bands+DOS 时，QE 输入文件的 5 个关键 flags（occupations / smearing / degauss / nbnd / nspin） 分别用什么？有没有一套「跑碱金属必备」的 cheat sheet 我能直接 hard-code 进 agent prompt 让 evaluator 总是能识别？
  5. dos_curve 评分用 normalized L2 RMSE + ±15 eV best-shift（EF 偏移已经独立 forgive），tolerance 0.1。
    但三版本平均只 0.02-0.25——形状本身和 GT 差太多。你们组里画 DOS 曲线，PBE 输出对什么参数最敏感？ 我猜测是这几个，请按重要性排：
    • (a) smearing 函数选择（gauss / mp / mv / fd）
    • (b) degauss / sigma 宽度（0.005 vs 0.02 vs 0.05 Ry 会让峰宽差 2-10x）
    • (c) NSCF k-mesh 密度（8×8×8 vs 24×24×24，过渡金属差异显著）
