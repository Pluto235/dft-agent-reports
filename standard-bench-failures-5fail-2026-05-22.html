2. 所有 timeout fail 的共通根因：FIX-6 estimator + bash hang

SOLVER · FIX-6 缺陷 核心 bug：pw_wall_estimator 公式 C × nat² × nk × (ecut/40)1.5 × multiplier × 1.5 里 C=4e-4 是未经任何 anchor 拍脑袋设的，对真实计算高估 10–90×。

在 agent/tools/bash.py:160–178 FIX-6 把 LLM 给的 timeout_s "抬"到 estimator 的估值（如果 estimator > LLM 值）。这 本意是防 LLM 低估，但实测 estimator 系统性过高 → timeout 抬到几十小时 → proc.communicate(timeout=est) 永不返回（pw.x 真实 wall < 真实 timeout，等不到自然结束也等不到超时） → bash tool 不 return tool_result → solver 转 transcript 卡死 → harness 4h wall 才砍。

FIX-9 改了什么

看 commit 59a8e80（FIX-9 fix(prompt): dos.x 协议跟 GT 同步到 tetrahedra）：system_prompt 里描述 dos.x 输入文件时强调 dos NSCF 要用 occupations='tetrahedra'。但 CoSb3 是窄 带半导体（PBE-near-metal）——FIX-3b 已经把它标了 pbe_near_metal=True， FIX-8 强制 scf/bands NSCF 用 smearing。问题来了：

  • bands NSCF：smearing（FIX-8 强制）
  • dos NSCF：tetrahedra（FIX-9 强制）
  • scf：smearing（FIX-8 强制）

这里其实没直接冲突（dos 和 bands 是不同 NSCF），但 dos.x 读 dos NSCF 产出后再用 tetrahedra 积分，而 dos NSCF 的 k-mesh 通常较稀，tetrahedra 在 窄带半导体 fermi 附近的 dos 计算容易抛错（tetrahedra 要求规整 dense Monkhorst-Pack k-mesh，含 kshift 时格式有 corner case）。我们没拿到 19:36 那次的 bash.log 来确认是不是 dos NSCF 那一步挂的（bash.log 文件不存在 = bash tool 第一个 pw.x 之前 hang 了，所以也无法 forensic）。

不完全成立的指控：FIX-9 与 19:36 失败的相关性很强（同一天 唯一的变量），但 bash.log 缺失阻止了直接因果断言。我们能 100% 锁定的是： 07:08 那条用 FIX-6/-7/-8 三件套拿了最高分 0.95，证明 FIX-8 单独看是正向； 10:04 上 FIX-9 后又挂了——要么是 FIX-9 真有副作用，要么是 FIX-6 在 dos NSCF 那步触发了和 Case A/B 同样的 estimator hang（CoSb3 24 原子，n_kpt 容易大）。
FIX-12 提议 (1) 在 advisor 给的 dos NSCF input 模板里，对 pbe_near_metal=True 的 material 强制保留 smearing（不走 FIX-9 的 tetrahedra 覆盖）—— GT 的 dos curve 比较是 broaden 后曲线，smearing 出来的 dos 也 能匹配。
(2) 端到端 audit：把 FIX-6/-7/-8/-9 单独 ablation 跑 5 次 CoSb3 / Ti3O5，看哪个 fix 单独 enable / disable 时回归。

turn 44 LLM 只产了 thinking（reasoning_content），没产 visible text，没产 tool_call。这是 deepseek 合法 的返回——thinking-only 模式下模型决定不 说话也不调工具，进入"在思考下一步该做什么"的状态。但 agent/loop.py 里的 message 拼装逻辑没考虑这种情况：

# agent/loop.py:183–202 — 凶手代码
asst_msg: dict[str, Any] = {
    "role": "assistant",
    "content": resp.text or None,   ← resp.text="" → None
}
if resp.tool_calls:
    asst_msg["tool_calls"] = [...]  ← tool_calls=[] → 不挂
if resp.reasoning_content:
    asst_msg["reasoning_content"] = resp.reasoning_content
messages.append(asst_msg)            ← 喂下一轮 API

if not resp.tool_calls:
    return {"incomplete": True, ...}

关键 BUG：messages.append(asst_msg) 在 return 之前 就执行了。所以即使 loop 退出，这条畸形 assistant message 已经写进 messages 列表。但 orchestrator 的 outer 流程在 verifier 触发后会用 executor.resume() 继续 loop（loop.py:160 注释提到 "continue-session" 路径）—— resume 时 client.call(messages=...) 就把上次留下的空 assistant message 也发出去了 → 400。

FIX-11 提议（具体行号） 在 agent/loop.py:186，把： 
"content": resp.text or None,
改为： 
"content": resp.text or (
    "(internal: thinking step, continuing without visible output)"
    if not resp.tool_calls else None
),
即：当模型既无 text 又无 tool_calls 时填一个占位 content，避免 API 拒绝。 另外在 orchestrator.resume() 入口处剪一下 messages 列表 末尾的畸形 assistant message（防御性）。这是 deepseek-v4-pro 的 thinking 模式独有特性，sonnet / haiku 不会触发；要在 README 标记 deepseek 兼容性。
