Bench Run Data 双向交换协议（规范）

这是什么：Pluto235（Shijie）和 weilin 两边各自部署了同一个 web 项目（fork 自 wanweilin/dft-agent-bench-web-fresh），各跑各的 bench 实验。本协议规定怎么把 一方机器上的 run 数据（state.db + runs_local/<run_id>/）打包传给另一方， 让对方的 web UI 也能看到这些 run 的 leaderboard / transcript 详情。

这不是什么：不是 step-by-step 操作手册。本协议是规范，配套已实现的脚本 一起用。人类同学读完 §2 数据流图 + §3 / §5 / §6 几张表，就能理解整个协议在做什么； 要动手照着 §10 的命令模板抄就行。

脚本：wanweilin/dft-agent-bench:tools/data_exchange/{export.py, import.py, README.md}。 PR #13 合并后落到 main； 合并前临时从 Pluto235/dft-agent-bench:feat/data-exchange-scripts 拉。

版本：2026-05-15 · v3（基于 shanhai → lambda 126 runs 的实跑验证修订； 原 2026-05-13 v2 内容已被覆盖，变更见 §11 changelog）

§1 双向数据流

同机器场景：共享归档目录就是中转 · 异机场景可换成 GH Release（见 §5）

本协议无方向性。weilin → shijie（即将发生）跟 shijie → weilin（已完成 2026-05-15， 126 runs 实跑通过）走完全一致的步骤，只是 source / destination 角色对调。脚本不区分方向， 靠命令行参数（--runs-local、--db、--out、--bundle） 显式声明。

§2 Bundle 形态

Source 端产出的 bundle.tar.gz 文件名约定： bench-bundle-<source>-<label>-YYYY-MM-DD.tar.gz， 其中 <source> 跟 §5 共享归档目录的子目录名一致（shijie / weilin）。 解开后目录树：

bench-bundle-<source>-<label>-<YYYY-MM-DD>/
├── manifest.json              # 元数据，见下方 schema
├── db_rows.jsonl              # 每行一个 run 的完整 state.db row（JSON 对象）
└── runs/
    └── <run_id>/              # 例如 2026-05-08T14-08-52Z_M1_T1_Si_vc_relax
        ├── report.json
        ├── harness.log
        ├── solver_stdout.log
        ├── solver_stderr.log
        └── workdir/
            ├── task_prompt.txt
            ├── transcript.json     # LLM 多轮对话
            ├── bash.log
            ├── result.json
            ├── lint.log
            ├── *.in / *.out        # QE 输入输出（文本）
            └── *.dat / *.dat.gnu   # bands / DOS 画图数据
            #  ↑ NOT included: pseudos/ (symlink), *.wfc*, *.save/, __pycache__/

manifest.json schema

字段类型说明
schema_versionint当前 = 1；升级协议时 bump
tierstringminimal / lossy / full，见 §10
exported_atstringISO 8601 UTC（如 2026-05-14T08:42:11Z）
labelstring人类可读标签，例如 "weilin baseline v1-stable 148 runs"
source.userstring跑 export 的 OS 用户名
source.hoststring主机名（仅用于溯源，不参与逻辑）
source.runs_localstring源端 runs_local/ 绝对路径（destination 用作 sanity check）
source.pseudo_dirstring源端 workdir/pseudos 软链指向的绝对路径——destination 端需要确认自己也有同名 pseudo 集
filterobject{kind: "where"|"run-ids"|"since", expression: ...}，溯源用
n_runsintbundle 包含的 run 数
db_columnslist[string]源端 state.db 的 runs 表实际列名（含 ALTER 加的）；用 PRAGMA table_info(runs) 获取
runslist[object]preview 列表，每项含 run_id / task_id / model / status / total_score

db_rows.jsonl 格式

每行一个 JSON 对象，键为 db_columns 列出的所有列，值即 sqlite row 内容。 顺序无要求。NULL 用 JSON null。日期 / 时间戳保留 sqlite 原始字符串。

§3 Source 端：过滤规则

对每个 run_dir，遍历所有文件按下表保留 / 丢弃：

路径模式规则说明
顶层（run_dir 直接子项）
report.json必保留评分、extracted_fields、audit_flags
harness.log / harness_stderr.log必保留主程序日志
solver_stdout.log / solver_stderr.log必保留solver 日志
workdir/ 内（视 tier 决定）
task_prompt.txt必保留原始题面
transcript.json必保留LLM 多轮对话，per-run 页面强依赖
bash.log必保留工具调用日志
result.json必保留solver 最终 submit 内容
lint.log必保留qe_input_lint 的 patch 记录（可能没有，OK）
*.in / *.outlossy 起保留；minimal 丢QE 输入输出（文本）
*.dat / *.dat.gnu / *.dat.raplossy 起保留；minimal 丢bands / DOS 画图数据（前端需要）
始终丢弃（无论 tier）
workdir/pseudos丢这是符号链接 → ~/bench/dft-agent-bench/pseudos/GBRV_USPP_PBE_v1.5。 naïve tar 会拷链接但目标在 destination 不一定存在。 **导入侧负责重链**（见 §4）。
*.wfc / *.wfc1 / *.wfc.tmp丢波函数二进制，单文件 20-470 MB；web 用不到
workdir/<prefix>.save/丢（整目录）电荷密度 + wavefunction 二进制；几百 MB
__pycache__/ / *.pyc丢scratch
任何其他符号链接丢避免悬空
tier=full 时除上面"始终丢弃"外全部保留注意 full 体积巨大（见 §9）

实测注解：在 shanhai 的 126 个 run 上跑 lossy tier，每个 run 平均 8–25 个文件、2–4 MB；老格式 run（无 workdir/，早期 schema 之前）会被 collect_run_files 自然跳过，不报错。state.db 里有但磁盘上没 run_dir 的 row（例如 cancelled 的 .pending/ 残留）仍会进 bundle 的 db_rows.jsonl， 只是 runs/<run_id>/ 是空的——destination import 时只 INSERT row，不抱怨。

§4 Destination 端：导入规则

DB 列改写

对 db_rows.jsonl 里每个 row，按下表处理后再 INSERT 到目标 state.db：

列名处理
run_dir必改：写成 destination 端的绝对路径 <dest_runs_local>/<run_id>
pid必改：设 NULL（源端进程在 destination 不存在）
owner保留源端值（leaderboard 不按 owner 过滤；保留可溯源）
created_at / started_at / finished_at保留（ISO UTC 字符串，跨机器有效）
其他所有列原样写入

§4.1 前置条件：destination web 必须先启动过一次

Destination 端的 state.db 列在协议 db_columns 之上还有 ~14 个 ALTER 加的列（solver_timeout_sec / cost_usd / owner / priority / agent_id / qe_call_count / qe_wall_sec / qe_wall_self_reported_sec / qe_wall_bash_cmd_sec / llm_wall_total_sec / input_cache_hit_tokens_total / input_cache_miss_tokens_total / qe_backend 等）。 这些列不在 backend/persistence/schema.sql 里，而是 backend/persistence/db.py 的 Database._init_schema() 在 web 启动时 逐条 ALTER 加进去的，失败容错（"duplicate column" 时静默吞）。

如果 destination 的 state.db 从来没被 web 碰过（比如只跑了 schema.sql 就直接 import），ALTER 列就不存在；source bundle 里这些字段会被 import 当作 schema drift drop 掉，日志里出现 WARN: source-only columns ...。

解决：destination 端 bash ~/web/start.sh 跑一次（哪怕跑完就停）， 让 web 把所有 ALTER 应用到 state.db，然后再 import。Web 跑着的时候 import 也 OK： sqlite WAL 模式并发读写无锁冲突。

Schema drift（两端列不一致）

两端 fork 可能有差异（一边 ALTER 加了列，另一边没加）。导入时：

  1. 用 PRAGMA table_info(runs) 拿 destination 端实际列名
  2. 跟 manifest.db_columns 求交集
  3. INSERT 只写两边都有的列
  4. source-only 列（dest 没有）：drop，打印警告
  5. dest-only 列（source 没有）：让 DB 用默认值或 NULL

pseudos symlink 重链

对每个导入的 run，在 destination 的 <run_dir>/workdir/ 下创建符号链接：

workdir/pseudos  →  <dest_pseudo_dir>

<dest_pseudo_dir> 由 import 脚本参数传入，通常是 ~/bench/dft-agent-bench/pseudos/GBRV_USPP_PBE_v1.5。 两端的 pseudo 集都跟 wanweilin/dft-agent-bench 同步，文件名一致； 如果 destination 缺该目录，import 脚本应 fail-fast 并提示。

冲突策略

如果某 run_id 在 destination state.db 已存在，按用户指定策略：

策略行为
skip（默认）跳过该 run（不动 DB、不覆盖 run_dir）
overwriteDELETE FROM runs WHERE run_id=? + rm -rf <dst_run_dir> → 再写新的
fail立即报错退出

§5 传输

§5.1 主推：共享归档目录（同机器场景）

shijie 和 weilin 在 lambda 同机器跑实验，最自然的传输路径是直接落盘在一个两人都可读写 的共享目录里：

/srv/shared/science/dft-agent-bench-data/
├── runs-archive-shijie/
│   └── bench-bundle-shijie-shanhai-migration-2026-05-15.tar.gz   ← 实例：126 runs · 17 MB
└── runs-archive-weilin/
    └── bench-bundle-weilin-<label>-YYYY-MM-DD.tar.gz             ← weilin 写到这里

每人有自己的 runs-archive-<source>/ 子目录。Source 端 export.py 的 --out 直接指向该路径下；destination 端 import.py 的 --bundle 指向同路径。Bundle 本身就作为归档保留——无需再清理 /tmp/。

权限：归档目录建议 chmod 2775（setgid + g+w），bundle 文件靠 umask 002 默认 664，两个用户都能读对方写的。如果一开始权限不对， chgrp -R <共用 group> /srv/shared/... 后再 chmod g+w。

§5.2 备选：GitHub Release（真正跨机器时用）

两边在不同物理机器、没有共享存储时改走 GH Release 中转。已验证：shanhai → lambda 迁移 2026-05-15 走的是这条路径，bundle 17 MB 上传 / 下载几秒钟。

共享仓 Pluto235/dft-agent-data-shares- 作为容器。Release 命名约定：<source>-<label>-YYYY-MM-DD，例如 shanhai-migration-2026-05-15 / weilin-baseline-148-2026-05-20。 两端用 gh release create / gh release download。文件 2 GB 上限对 lossy（< 50 MB）绰绰有余；full tier 要切片。

§6 反向使用（weilin → shijie）

协议对称。weilin → shijie 跟 shijie → weilin 共用同一对 export.py / import.py，只是命令行参数对调：

参数shijie → weilin（已完成）weilin → shijie（即将做）
Source 端 --runs-localshijie 的 runs_local 路径weilin 的 runs_local 路径
Source 端 --dbshijie 的 state.dbweilin 的 state.db
Source 端 --out/srv/shared/.../runs-archive-shijie/.../srv/shared/.../runs-archive-weilin/...
Destination 端 --bundle/srv/shared/.../runs-archive-shijie/.../srv/shared/.../runs-archive-weilin/...
Destination 端 --db / --runs-localweilin 的shijie 的
Destination 端 --pseudo-dirweilin 自己的 GBRV 路径shijie 自己的 GBRV 路径

Manifest.json 里的 source.user / source.host 自动反映谁是 source， 无需手动配置。§10 给出 weilin 这次要抄的完整命令模板。

§7 边界条件 / 注意事项

情景处理
源端 state.db 有 ALTER 加的列，destination 没有（schema drift） §4 流程已处理：dest 无的列 drop 并警告
某 transcript.json > 10 MB（卡死循环的 run） 原样打包；destination web 渲染可能慢但不会崩。脚本可选实现 --max-transcript-mb 截断
跨时区 created_at 显示 DB 存 ISO UTC 字符串（如 2026-05-08T14:08:52Z），前端按本地时区渲染。不是 bug
destination 缺 <dest_pseudo_dir> import 脚本 fail-fast，提示先 git pull 同步 dft-agent-bench 仓
bundle 里某个 run_id 没有对应 runs/<run_id>/（损坏） skip 该 row，记到失败列表
destination web 已经在运行，DB 正在被读 sqlite WAL 模式下并发读写 OK；如果有写锁冲突，import 端可以重试或停 web 后再 import
tier 是 minimal 但 destination 想画 bands / DOS 图 展示会缺曲线数据。需要重新让 source 端用 lossy 重打一次
transcript.json schema 跨 v1 / v2 / v4 web/backend/services/transcript_adapter.py 已兼容；无需迁移
destination web 从来没启动过、state.db 缺 ALTER 列 见 §4.1；不致命但会 drop 列。Destination 先 bash ~/web/start.sh 一次再 import
共享归档目录没写权限 export.py 在写 --out 时 fail-fast。先 chmod 2775 归档目录、统一 group ownership
同机器两个用户 runs_local 路径不同 无所谓：import 的 --runs-local 写 destination 自己的路径即可。manifest 里 source.runs_local 仅做溯源不参与逻辑；run_dir 列由 import 改写成 destination 路径（协议 §4）
state.db 里某行 run_dir 在磁盘上不存在（cancelled .pending/ 残留） Export 时该 row 仍进 db_rows.jsonl，但 runs/<run_id>/ 不会创建。 Import 时 INSERT row 但跳过文件拷贝，日志标 db-only

§9 附录：Tier 体积对比

Tier单题大小（估算）148 题估算实测（shanhai 126 runs · 2026-05-15）用途
minimal~200 KB~30 MB未实测leaderboard、transcript、看分数
lossy（默认）2–4 MB~500 MB104 MB 解压 / 17 MB tar.gz+ 能带 / DOS 曲线、SCF 收敛过程
fullmedian 1 MB / max 7.7 GB~36 GB未实测bit-exact 复现；GH Release 单文件 2 GB 上限须切片

实测 lossy 远小于估算的原因：QE .in / .out 和 .dat 主要是 ASCII 文本，gzip 压缩率 ~6×；早先 ~500 MB 估算是按文件原大小算的偏保守值。
Full tier 想发的话按 task group（M1_T1 / M1_T2 / M2_T1 / M2_T2）切成 4 个 bundle 分别上传 release；destination 端循环 import 即可。

§10 已验证的工作流（weilin 抄这一节）

下面是 weilin 在 lambda 上把自己的 run 数据发给 shijie 的完整命令模板，按顺序在 lambda 的 shell 里跑。

步骤 0：一次性拉脚本

PR #13 合并前临时从 shijie 的 fork 拉：

cd <weilin 的 dft-agent-bench 仓>
git remote add shijie-fork git@github.com:Pluto235/dft-agent-bench.git 2>/dev/null || true
git fetch shijie-fork feat/data-exchange-scripts
git checkout shijie-fork/feat/data-exchange-scripts -- tools/data_exchange
ls tools/data_exchange/   # 应该有 export.py / import.py / README.md / __init__.py

PR merge 之后改成 git pull upstream main 即可，不再需要 shijie-fork remote。

步骤 1：选要分享的 run + Export

三种过滤方式三选一，或不传过滤全量发：

~/miniconda3/envs/tritondft/bin/python tools/data_exchange/export.py \
  --runs-local <weilin 自己的 runs_local 绝对路径> \
  --db         <weilin 自己的 state.db 绝对路径> \
  --tier lossy \
  --label "<一批 run 的人话标签，例如 baseline-148>" \
  --out /srv/shared/science/dft-agent-bench-data/runs-archive-weilin/\
bench-bundle-weilin-<label>-YYYY-MM-DD.tar.gz \
  # 可选过滤（三选一）：
  # --filter "status='done' AND total_score >= 0.5"
  # --run-ids /path/to/ids.txt
  # --since 2026-05-01
  # 可选预览：先 --dry-run 看 run 数和大小估算

跑完命令最后一行会打 bundle written: ... (N MB)。

步骤 2：通知 shijie

把 bundle 文件名告诉 shijie。Shijie 这边一条 import 命令就把数据合并进他的 web 的 state.db（参考 shanhai → lambda 已经跑过的一遍）：

~/miniconda3/envs/tritondft/bin/python ~/projects/bench/tools/data_exchange/import.py \
  --bundle      /srv/shared/science/dft-agent-bench-data/runs-archive-weilin/\
bench-bundle-weilin-<label>-YYYY-MM-DD.tar.gz \
  --db          <shijie 在 lambda 上 web 的 state.db 路径> \
  --runs-local  <shijie 在 lambda 上 web 的 runs_local 路径> \
  --pseudo-dir  ~/projects/bench/pseudos/GBRV_USPP_PBE_v1.5 \
  --on-conflict skip

Import 跑完后 shijie 这边 web 重启（或直接刷新页面，sqlite WAL 模式下不用重启）就能在 leaderboard 看到 weilin 发来的 run。

步骤 3：归档自动保留

Bundle 留在 /srv/shared/science/dft-agent-bench-data/runs-archive-weilin/ 作为永久归档。后续两边任何人想回看"上次 weilin 发的是哪 N 个 run"，直接 tar xzf ... -O manifest.json 就能看到 manifest 里的 label / filter / runs preview，无需再问。
