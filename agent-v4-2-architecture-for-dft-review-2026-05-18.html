1 · 背景：v1 baseline 是什么，为什么不够

v1 是我们最早能跑通 bench 的 solver。整个 agent 就是一个 LLM，给它 4 个工具 （bash / write_file / read_file / submit） 和一段 42 行的通用系统提示词，然后让它读 task prompt、自己想、自己写 QE 输入、自己跑、 自己判断收没收敛、自己 submit。形式上是经典的 ReAct 循环（think → act → observe → think → …）。

v1 像一个临时雇来的远程同事，啃过 QE 文档但没真做过项目。你把题目甩给他， 他会试着做；但你不在旁边，他怎么做、查没查、什么时候放弃，全凭他自己。

这种最小架构在 6 道 M1_T1（单元胞 vc-relax，Si / NaCl / Cu 这种）上能跑出 0.7+ 的均分， 但只要难度上来，就会暴露几类系统性失败模式：

v1 暴露的失败模式物理表现 / 后果
vc-relax 用 QE 默认松阈值（press_conv_thr=0.5 kbar、 forc_conv_thr=1e-3 Ry/Bohr、etot_conv_thr=1e-4 Ry） cubic 系统的弛豫结构漂移 ≥0.01° 或 ≥0.05% → pymatgen SpacegroupAnalyzer 把它降级为 trigonal/orthorhombic，空间群分扣 0.25。实测 ~25% miss 率。
SCF 不收敛 → 死循环 LLM 反复改 mixing_beta / 加 nstep，2 小时墙时间用完，submit 不出来。
metal 用了 occupations='fixed' SCF 振荡或假收敛；或者 insulator 用了 smearing 把 gap 抹平。
大单胞 bands_dos 直接上来跑 bands 结构没先 vc-relax，bands 结果用错初始几何；或者 PBE+USPP 用 ecutrho=4×ecutwfc （NC pseudo 的默认）→ 带隙漂 0.2–0.5 eV。
bash 里 pip install / curl bench audit 直接判 0 分（评分要求纯离线运行）。
LLM 自己说"我做完了" 漏交付字段 / 提交了未收敛的 .out / 给的路径根本不存在。

这些不是"LLM 笨"的问题，而是 v1 架构里缺三层保险： 没人提前过滤输入（→ qe_input_lint）、没人在执行完之后独立检查物理（→ verifier）、 没人在出错时决定该重试还是放弃（→ orchestrator 状态机）。v4.2 就是把这三层都补上。

2 · v4.2 一图概览：三角色 + 两 ledger

v4.2 的核心结构可以一图概括：

图 1 · v4.2 顶层流程（Phase 1 preflight + Phase 2 三角色循环 + retry / replan 反馈）

三个 LLM 角色分工严格：

  • Planner：只看 task prompt，输出一个 PlanSpec（步骤 DAG，每步带 kind / deps / expected_outputs）。不读文件、不执行。
  • Executor：拿到单个 step 的 prompt（base QE prompt + step_kind addon）， 用 bash 等工具实际跑 QE，submit 该 step 的产物路径。
  • Verifier：用 step_kind 专属 checklist 检查 executor 产出的文件， 输出 pass / fail + diagnosis。diagnosis 是 replan 时 planner 唯一的反馈渠道。

两层 ledger（Magentic-One 风格）把所有状态序列化到 orchestrator.json： TaskLedger 装初始 plan + 重规划历史；ProgressLedger 装每个 step 的 attempt 列表（含 executor session、verifier diagnosis、orchestrator 决策）。 任何 bench run 跑完，整条 trace 都能复盘。

v1 是"一个人闷头干"；v4.2 是"项目经理（planner）写 spec，工程师（executor）实施， QA（verifier）按物理检查清单验收，工头（orchestrator）拿到 QA 报告决定返工还是改方案"。 每个角色都能独立换模型——bench 现在 planner / executor 都是 Claude Sonnet，verifier 用更便宜的 Haiku。

3 · v4.2 vs v1 ─ 每个新增组件解决什么具体问题

下表把 §1 那些 v1 失败模式，逐一映射到 v4.2 的应对机制；并标注它在哪一层 （prompt = 写进系统提示靠 LLM 自觉；hook = 代码强约束， LLM 没法绕过；verifier = 事后检查；orchestrator = 状态机）。

v1 失败模式v4.2 的应对在哪一层
vc-relax 阈值太松 → 空间群退化 qe_input_lint 在 pw.x spawn 之前拦截 input 文件， 强制 press_conv_thr ≤ 0.05、forc_conv_thr ≤ 1e-4、 etot_conv_thr ≤ 1e-5；LLM 自己写多松都会被收紧。 hook（代码强约束）
SCF 死循环 → 跑墙时间 每个 step ≤2 次 retry；耗尽后 ≤2 次 replan； 最终 give-up 写 incomplete。 orchestrator 状态机
metal 用 fixed occupations config_advisor 在 planner 启动之前先确定性分类（METAL_ELEMENTS 硬编码表 + 化学式正则 + "强电负性元素 → 化合物"），LLM 只做交叉验证； 结果注入 planner / executor 的 prompt 和 .config_advice.json。 preflight 子 agent + prompt
USPP 用 ecutrho = 4×ecutwfc qe_input_lint Rule C：检测到 pseudo 家族含 USPP/GBRV/PAW，自动把 ecutrho 改成 10×ecutwfc。 hook
bands NSCF 用了 calculation='bands' qe_input_lint Rule B：发现 K_POINTS crystal_b + calculation='bands' 组合，自动改成 'nscf'。原因是 bench evaluator 要从 highest occupied, lowest unoccupied level 这行抓 VBM/CBM， QE 只在 nscf 模式下写这行。 hook
大结构直接 bands → 几何没松弛 vc_relax preflight：T2（bands_dos）任务在 planner 跑之前先做一次松弛， 把松弛后的几何 + 能量塞进主 executor 的 prompt（避免主 loop 把预算耗在结构上）。 preflight 子 agent
bash 网络命令 → audit 0 分 bash_guard 黑名单 curl / wget / ssh / pip install（除非证明本地确实没装）； pw.x 后台化（& / nohup / setsid）也被拒。 hook
LLM 自报"做完了" → 漏字段 / 提交未收敛 Verifier 按 step_kind 的 verifier_checklist 检查每个 expected_output 的文件存在 + ≥1024 B + JOB DONE.（log 文件）+ 关键物理标志 （比如 vc-relax 必须有 bfgs converged in）。 verifier LLM + 确定性检查
bench fermi-slice 漏算金属（已知物理 vs 抽取错配） physics_sanity step（bands_dos 自动注入）：拿 config_advisor 的高置信 is_metal 标签 vs bench evaluator 抽取出的 gap_type / band_gap_ev， 失配（已知 metal × 抽取出 >0.5 eV gap）直接触发 replan，要求加大 nbnd / 加密 k-path。 orchestrator 自动注入 + 确定性检查

注意整张表里只有一条用了"靠 prompt 自觉"（config_advisor 的物理结论），其他全是 hook / 状态机 / 事后验证。这是 v4.2 vs v1 最核心的哲学差异：v1 把 DFT 知识"告诉" LLM 让它自己用；v4.2 把同样的知识变成代码层强约束，LLM 想绕都绕不过去。

4 · 四个 DFT 专家最该审的「硬编码决策」

下面四个机制，每个都把具体的 DFT 物理判断写死在了代码或 prompt 里。 我们想请你看：这些默认值 / 判定规则，物理上对吗？漏了什么？

§4.1

Knowledge injection — system prompt 里塞了什么 DFT 知识

做了什么：v4.2 的 executor system prompt 由 agent/system_prompt.py 的 build_qe_system_prompt() 拼出，里面硬编码了 QE 工作流的"潜规则"。每个 step 又叠加该 step_kind 的 addon （agent/step_kinds/*.py 的 system_prompt_addon()），加上从 config_advisor 拿到的 QE CONFIGURATION ADVISORY 块。

注入的具体内容（节选最物理相关的部分）：

  • 赝势路径纪律：必须 pseudo_dir = './pseudos/'，否则 pw.x 落到 容器内 /root/espresso/pseudo/ 不存在，Error in routine readpp。 这是新手最常踩的坑——专门写进 prompt。
  • vc-relax 三阈值（重复写在 §4.4 详述）： press_conv_thr=0.05 / forc_conv_thr=1.0d-4 / etot_conv_thr=1.0d-5， 并在 prompt 里给出完整的 vc-relax 输入模板。
  • bands_dos 工作流 canonical 参数：bench T2 GT 用了 ecutwfc=80, ecutrho=800 (10×, GBRV USPP 标准), kmesh ≈ ceil(50/a_Å) per axis, min 8×8×8。 Evaluator 用 ±0.1 eV 容差打分；参数松一点就会确定性扣分。
  • bands NSCF 必须 calculation='nscf' 而不是 'bands'： QE 只在 nscf 模式下写 highest occupied, lowest unoccupied level (ev): ... 这行，evaluator 靠它抓 VBM/CBM。
  • DOS NSCF 用 occupations='tetrahedra'；dos.x 用 deltaE=0.01 eV, ngauss=0, degauss=0.0015 Ry (=0.02 eV) 来匹配 GT 的能量栅格和高斯展宽， 不然 DOS shape RMSE 会跪。
  • 晶格 6 标量 → CELL_PARAMETERS 必须用 pymatgen Lattice.from_parameters， 禁止手算（rhombohedral 易把 a*sqrt(2/3) 错写 a*sqrt(8/3) → 翻倍体积 → metal/insulator 都判错）。
  • k-path 必须用 seekpath.get_path，不允许手挑高对称点。
  • occupations 选择：semiconductors/insulators → 'fixed'； metals → 'smearing'（mv, degauss=0.02）。

我们的理由：这些都不是 QE 文档第一页就能看到的，而是"做过项目才知道" 的潜规则。每一条都对应过去 v1 / v3 / v4.1 上至少踩过一次的失分 case（详见 入门教程 §6 / §15）。 但 DFT 圈一定还有"大家都这么做但论文不写"的细节我们没注入——这是想请教你的（→ §6.1）。

§4.2

Verifier 的物理 sanity checklist（按 step kind）

做了什么：每个 step_kind 模块都导出一个 verifier_checklist() 函数，verifier LLM 拿到这个 checklist 作为 system prompt，再去 read_file 检查产物。 checklist 是物理 + 文件两层。

各 step kind 的 checklist（节选关键项）：

step kind必查项失败诊断举例
scf 文件存在 + ≥1024 B；log 最后 64 KB 含 JOB DONE.； 不含 convergence NOT achieved；out/<prefix>.save/ 含 charge-density.dat（或 .hdf5）且 size>0 "out/si.save 为空 → 电荷密度没写出来"
vc_relax JOB DONE + bfgs converged in 或 End final coordinates； 无 maximum number of steps has been reached； 最终 total stress ≤ 0.05 kbar；声明 cubic 的体系， 最终 a/b/c 在 0.01 Å 内一致 "BFGS 未收敛——bench 会按未收敛结构打分"
relax JOB DONE + bfgs converged in / End final coordinates； 无 convergence NOT achieved 同 vc_relax
bands 所有 *_out 含 JOB DONE；*_dat（bands.dat）≥1024 B 且像 ASCII 数据列 "bands.dat 只有 200 B → bands.x 没正确跑完"
dos nscf log 含 JOB DONE；dos.dat ≥1024 B 是数值列；无 Error in routine "dos.dat 全 0 → fildos 路径写错或 dos.x crash"
collect 纯文件操作：每个 expected_output 存在 + 非空（合法的"无 LLM 行为"的步骤） "目标路径没生成——是 cp 命令路径写错？"
physics_sanity 纯 Python 跑（不调 LLM）：拿 config_advisor 的 is_metal + source 高置信 vs bench evaluator 抽取出的 gap_type / band_gap_ev； 失配（metal × extracted gap>0.5 eV，或 insulator × extracted 'metal'）→ verdict='fail' + replan "已知 Rb 是金属但 fermi-slice 抽出 1.2 eV gap → 加大 nbnd 后重做 bands NSCF"

📌 v2 update（2026-05-20）：physics_sanity 的规则集已经从初版的 4 条 (Rules A–D) 扩到 6 条，每次 run 都并行评估，最终 verdict 取最高严重度：

规则触发处置
A · metal × extracted gap>0.5 eVconfig_advisor 高置信 metal，但 bench 抽到 gap fail + replan（提示加大 nbnd + 改 k-path 覆盖 Fermi 穿越点）
B · insulator × extracted metalconfig_advisor 高置信非 metal，但 bench 抽到 metal fail + replan（提示 NSCF 应该 occupations='fixed' + 增 nbnd）
C · 已知 wide-gap × extracted gap<0.05 eVadvisor rationale 含 oxide/fluoride/wide-gap 但抽到几乎为 0 soft_warn（PBE 系统性低估，不 retry）
D · 低置信 advice 出现物理矛盾非 deterministic source 上发生 A 或 B 情形 soft_warn（advice 不够硬，不强 replan）
E · 磁性矛盾（新） task spec 声明了 main_magnetic_element 但 SCF 最终 total_magnetization < 0.1 µB fail + replan（提示 nspin=2 + starting_magnetization(X) 起点）
F · vc_relax 残余应力（新） preflight relax.out 最末 P 块的 |P| > 5 kbar（≈0.5 GPa） soft_warn（PBE 误差有时确实留 0.5 GPa；写进 notes，提示 press_conv_thr 或 cell_dofree）

Rules E + F 还在 working tree（未 commit），但 tests 已经写完；Rules A–D 已经 merge 在 731a626。这两条专门请教 §6.2：阈值取得对吗？还该补哪些 deterministic sanity？

我们的理由：v1 让 LLM 自报"做完了"，经常漏字段或提交未收敛的结果；v4.2 把"做完了"的判断从 LLM 自评挪到了独立 verifier + 物理 checklist。 checklist 现在大多还是文件 / 字符串级别的判断，"真物理 sanity" （比如 SCF 形式上收敛但电荷密度异常、PBE 算金属带隙不为 0 但很小）覆盖得还不够厚—— 这是想请教你的（→ §6.2）。

§4.3

Step kinds 的切分

做了什么：v4.2 把 DFT 工作流切成 7 种 step kind，每种 一个 Python 模块（agent/step_kinds/*.py），导出 system_prompt_addon + tool_allowlist + verifier_checklist 三个钩子。Planner 出 plan 时只能从 这 7 种里挑。

kind职责
scf自洽场——给定结构 → 电荷密度 + scf.out
vc_relax变胞弛豫——晶格 + 内坐标全松，三阈值强制收紧
relax固定胞内坐标松弛——很少用，通常 vc_relax 替代
bandsk-path bands NSCF + bands.x post-processing，依赖上游 SCF charge density
dos密集 k-mesh NSCF + dos.x，依赖上游 SCF
collect纯文件操作（cp / rename / 合并数据），禁止调 pw.x / bands.x / dos.x
physics_sanitybands_dos 任务自动注入的尾步——物理 vs 抽取的一致性检查，纯 Python 不调 LLM

我们的理由：把工作流"切片"的好处是每个切片可以独立写 prompt 和 checklist——SCF 的 prompt 关心电荷密度，bands 的 prompt 关心 k-path 和 nbnd，互不污染； verifier checklist 也按切片定制。新增 step kind = 加一个模块 + 注册，orchestrator 不动。

但 bench 题目今年要扩到 phonon / 缺陷 / 表面能 / DFPT；当前 7 种是否够、该不该新增、 或者粒度不对（比如 nscf 是否该独立成一类？）——是想请教你的（→ §6.3）。

§4.4

qe_input_lint 的默认值表 + 处理策略

做了什么：每次 BashTool 准备 spawn 一个 pw.x 调用， 都会先把对应的 input 文件交给 agent/tools/qe_input_lint.py 跑 4 条独立规则； 任何一条触发都会把 input 改写到磁盘，并把 --- qe_input_lint --- diff 块返回给 LLM （让 LLM 看到自己被改了什么，下次别再写错）。

规则触发条件动作默认值 / 理由
A · vc-relax 收敛阈值 &CONTROL.calculation = 'vc-relax' 强制 min(LLM 值, 紧默认)——只能更紧不能更松 press_conv_thr=0.05（kbar；QE 默认 0.5）
forc_conv_thr=1.0d-4（Ry/Bohr；QE 默认 1e-3）
etot_conv_thr=1.0d-5（Ry；QE 默认 1e-4）
理由：cubic 系统的 SpacegroupAnalyzer 对漂移 ≥0.01° / 0.05% 就降级
B · bands NSCF 模式 calculation='bands' 且 K_POINTS crystal_b/tpiba_b 改成 calculation='nscf' QE 只在 nscf 模式下写 highest occupied / lowest unoccupied 行； evaluator 靠这行抓 VBM / CBM
C · USPP ecutrho 比例 pseudo 家族含 USPP / GBRV / PAW 且 ecutrho/ecutwfc < 9 把 ecutrho 改成 10 × ecutwfc GBRV USPP 标准比例 ~10×；4–8× 会让带隙漂 0.2–0.5 eV / 30 Ry ecutwfc
D · occupations advisory SCF / bands-NSCF 的 occupations 和 config_advisor 推荐不一致 仅警告，不自动改（advisor 在 Mott 绝缘体等边界 case 可能错） config_advisor 推荐：metal → smearing（mv, degauss=0.02）； insulator → fixed
E · K_POINTS crystal_b 数据行校验
v2 新增 (uncommitted)		 bands NSCF 模式且 K_POINTS crystal_b / tpiba_b 检查每段路径数据行格式：先是段数 N，随后 N 行 kx ky kz weight（4 列）； 发现段数与行数不符 / 行少于 4 列 / weight 非整数 → 拒绝 spawn pw.x， 把诊断回写给 LLM 让其重写 input 典型坑：LLM 把 seekpath 输出粘错 + 漏写 weight 列；或者把高对称点标签也算进段数。 这条之前在 Si bands 案例上踩过，pw.x crash 给 "found ... lines, expected 8" 但 LLM 看不懂错误码

处理策略的两条核心：

  1. min(LLM 值, 紧默认)——只收紧不放松。我们的假设是"如果题目本身能容忍松一点， 那紧一点也不会更差，只是多花 1-2 个 BFGS 步"。是否成立请教 §6.4。
  2. 幂等 + 把 diff 回写给 LLM——不偷偷修改，让 LLM 在下一回合看到自己刚才写的 跟最终落盘的差，逐步学会自己写对。
§4.5

experience library — 自动累积的"方法学教训"（v2 新增）

做了什么：v4.2 在 05-19 加了一个全新的子系统 （commit ba9fba2），让 agent 把每一次"先失败、后修好"的 step 当成一次教训 攒下来，下次同类 step 启动时把它注入对话上下文。范式上是 Reflexion（Shinn et al. 2023） 的一个工程化变种，针对 DFT 任务做了"方法学 only / 不缓存答案"的硬约束。

跟 §4.1–§4.4 不一样的地方：这一节里"硬编码"的不是 DFT 知识本身，而是 "什么算合法的教训"的边界——具体 DFT 教训内容是 agent 自己学出来的。

experience library 闭环：fail → recover → reflection → JSONL → 下次 pre-seed。 每个 step_kind 一条独立 JSONL；scope 决定哪些教训会被取出来注入。

为什么"方法学 only"

这个子系统最大的风险是无意中把答案缓存起来——比如有人写了一条 "Si 体系 ecutwfc 用 80 Ry 才能让 gap 收敛到 0.012 Ha"，下次再跑 Si 直接命中，看起来 "学会了"但本质是答案缓存。Bench 评的就是 agent 通用 DFT 能力，不是查表能力。

所以 validate_lesson() 在写入前用 regex 三层过滤（顺序很重要）：

def validate_lesson(lesson, required_keys):
    # 1. shape: {trigger, action, rationale}，每段 ≤ 200 chars
    # 2. 先 mask QE keyword RHS，让合法的 `mixing_beta=0.3` 不触发 unit 检查
    text   = " || ".join(lesson[f] for f in ("trigger","action","rationale"))
    masked = _QE_KEYWORD_RE.sub(lambda m: f"{m.group(1)}=__masked__", text)
    # 3. 在 masked 文本上拒绝以下任一：
    #    - 任何 118 元素符号（word-boundary，含同位素后缀）
    #    - 任何 required_keys 字面 token（bands_data / gap_type / nscf_dos_out…）
    #    - 任何 `<number><unit>` 形状（eV / GPa / Å / Ry / bohr / kbar）
    return (no_violations, reasons_list)

QE keyword 白名单（ecutwfc / mixing_beta / conv_thr / occupations / nbnd …） 先把 ecutwfc=80 这种合法的方法学陈述屏蔽掉，剩下的文本里再看到 任何 number+unit 都视为答案泄漏。

scope 维度怎么定的

每条教训写入时带 scope tag；下次 load_lessons 按 scope 从同 step_kind 的 JSONL 里挑出 top-5 最新（去重）。当前 scope 由这 4 个维度组成：

scope 维度枚举值为什么选它
task_kind vc_relax（M1_T1）/ bands_dos（M1_T2） 不同任务类型对参数敏感度差异最大；T1 教训不该污染 T2 的 NSCF 设置
crystal_system cubic / tetragonal / orthorhombic / hexagonal / trigonal / monoclinic / triclinic cubic 系统对 SpacegroupAnalyzer 漂移最敏感；六角 / 三角对 c 轴 ecutwfc 异常更敏感； 分对称族存放避免"立方体系下的教训用到 triclinic"
n_atoms_bucket ≤2 / 3-8 / 9-32 / >32 大单胞需要更细 mixing_beta / 更大 diago_thr_init；小单胞反之。 按数量级分桶（不是连续值）避免桶过细 → 命中率太低
step_kind scf / vc_relax / bands / dos / collect / physics_sanity 每个 step_kind 独立 JSONL；vc_relax 的教训不会污染 bands 的 LLM 上下文

注入策略是同 4 维全匹配优先；如果 0 条，回退到 {task_kind, step_kind} 两维 loose 匹配。返回时按 (trigger, action) 去重，最多 top-5 注入。

什么时候触发 reflection_writer

只在"先 fail 后 succeed"的 step 上触发。每个 step 一旦 verifier 给出 'pass'， orchestrator 看这个 step 的 attempts 历史：如果存在 ≥1 个 verifier='fail' 的 attempt， 就调 reflection_writer subagent（独立 LLM session，便宜档），喂给它：

  • failed attempt 的 verifier diagnosis（"BFGS 在前 3 步震荡"）
  • fail 和 pass 之间的 write_file diff（同路径下 LLM 改了什么； 硬上限 4 KB，超出截断）
  • system prompt 里 hard rules：禁止写元素符号 / required_keys / number+unit （regex 验证器会再 catch 一次，subagent 只是先一道防线）

Reflection subagent 调一个 submit_lesson tool（schema 是 trigger / action / rationale / decline + reason）；如果它觉得这次 recover 没什么可教（比如 attempt 1 死在打错文件名），就 decline=true，不写。空的 library 比毒的 library 强 是这一节的核心约束。

storage 形态

每个 step_kind 一个 append-only JSONL：

// experience/vc_relax.jsonl 中的一行（示例）
{
  "id": "a3f4b1e8",
  "scope": {"task_kind": "vc_relax", "crystal_system": "cubic",
            "n_atoms_bucket": "≤2", "step_kind": "vc_relax"},
  "lesson": {
    "trigger":   "vc-relax 前 3 步 force 在 1e-2 量级震荡，BFGS 一直拒接受",
    "action":    "把 mixing_beta 从默认 0.7 调到 0.3，并加 electron_maxstep=200",
    "rationale": "强电子相关或大原子序数时 charge mixing 易振荡；
                  小 beta 牺牲收敛速度换稳定性"
  },
  "written_at": "2026-05-19T14:23:11.582901+00:00"
}

永远 append，从不删；id 是 4 字节 hex；written_at 微秒精度（一秒内多次写入也能稳定排序）。仓库一致性靠 newest-first + dedup 投影实现，不靠 删除旧条目——这样不同节点 / 不同 run 的 library 可以独立成长，事后做 union merge。

同期顺手做的几个小改（不单独成节）：
  • max_replans 默认 2 → 5（eb06826）——给 planner 更多 recovery 预算。 早期 v4.2 跑 standard27 时观察到 2 replan 后还在挣扎的 case 占失败的 ~15%。
  • 所有 subagent (executor / verifier / reflection_writer) 的 LLM 调用都双写到 主 workdir/transcript.json（7368bc5）——以前每个 subagent 自己写自己的 transcript.json，web 端 cost reconciler 看不到总 token 花费；现在统一审计。
  • Planner 在 submit_plan 里声明的 bonus keys（required keys 之外 的额外输出）现在也会出现在 result.json 里（250ad38）——以前被 audit 过滤掉，导致 bench 端"没看见"agent 多交付的字段（比如自检图）。

5 · 一个真实 trace 示例 · M1_T1 Si vc-relax

下面是 2026-05-18 跑的一个 Si vc-relax 任务，完整 trace 在 /srv/shared/science/web/runs_local/2026-05-18T11-56-54Z_M1_T1_Si_vc_relax/。 完整时间 408 s，总分 1.0。从 task_prompt 到 result.json 整条决策链都可以复盘。

展开节选：planner → executor → verifier 各 1 turn 
[Phase 1] config_advisor:
  formula="Si", elements=["Si"], pseudo_family="SSSP/efficiency (USPP)"
  is_uspp=True
  is_metal=False               # Si 不在 METAL_ELEMENTS 表里
  occupations="fixed"          # 半导体 → fixed
  ecutrho_factor=10.0          # USPP 标准
  source="deterministic"

[Phase 2 / Planner turn 1]
  调用 submit_plan:

      { id: "vc_relax_1", kind: "vc_relax", deps: [],
        expected_outputs: { "vc_relax_out": "vc_relax_1/vc-relax.out" } }
    ]

[Phase 2 / Executor (vc_relax_1) turn 3]  <-- LLM 写完 input 后第一个 pw.x 调用
  bash: pw.x -in vc_relax_1/vc-relax.in > vc_relax_1/vc-relax.out 2>&1
  pre-spawn hook: qe_input_lint Rule A 触发
  ──── auto-injected by qe_input_lint ────
  file: vc-relax.in
  - press_conv_thr = (missing)   →   press_conv_thr = 0.05
  - forc_conv_thr  = 1.0d-3       →   forc_conv_thr = 1.0d-4
  reason: vc-relax requires tight thresholds (v3 hard rule)
  ────
  → input 被改写后再 spawn pw.x

[Phase 2 / Verifier (vc_relax_1)]
  read_file vc_relax_1/vc-relax.out (tail 64 KB)
  ✓ "JOB DONE." 存在
  ✓ "bfgs converged in 4 steps" 存在
  ✓ 最终 stress = 0.02 kbar < 0.05
  ✓ cubic 系统：a = b = c = 5.4307 Å（差异 < 0.0001 Å）
  → verdict: pass

[Final] result.json:
  task_outputs = { "vc_relax_out": "vc_relax_1/vc-relax.out" }
  total_score  = 1.00
  audit_flags  = all pass (qe_wallclock_ok, no_network, no_background_pw, ...)

这条 trace 完整展示了三层保险一起干活的样子：preflight 提前定 occupations = fixed（Si 半导体），qe_input_lint hook 在 spawn 之前补上 LLM 漏写的收敛阈值， verifier 事后查物理一致性。任意一层失守，其他两层仍能兜住——这就是 v4.2 相比 v1 的"纵深防御"。

6 · 请教 DFT 专家的具体问题

下面 4 个问题对应 §4 的四节。任何一条你觉得"不对"、"太松"、 "太紧"、"漏了 X 物理场景"，都请直接告诉我们。

§6.1 · Knowledge injection 够不够

请审 §4.1 列出的注入项：

  • 三阈值（vc-relax）、USPP ecutrho ≈ 10×、bands 必须 nscf 模式、DOS 用 tetrahedra、 occupations 选择规则——有没有写错的？
  • DFT 圈"大家都这么做但论文 / QE 文档不写"的潜规则， 我们漏了哪些？例如：磁性体系的 starting_magnetization 起点、d/f 电子体系的 Hubbard U、 LDA+U 时的 starting_ns_eigenvalue、SOC 起点是不是要 PAW 而不是 USPP……
  • bands_dos canonical 参数（ecutwfc=80, ecutrho=800, k-mesh ≈ ceil(50/a)） 对每类材料是否合适？氧化物 / 二维材料 / 重元素 / 磁性体系，是不是该用不同的 canonical？
  • 赝势选择本身——bench 用 SSSP efficiency（混合 USPP / PAW / NC）； 你建议给 agent 哪些"该用 PAW 而不是 USPP"的硬规则？

§6.2 · Verifier 的物理 sanity check 够不够厚

请审 §4.2 那张 checklist 表：

  • 现在的检查大多是"文件存在 + 字符串匹配"。哪些真正的物理失败模式 我们没覆盖？例如：
    • SCF 形式上 JOB DONE. 但能量明显偏离（比如比经验值低 / 高几个 eV/atom）
    • vc-relax 最终 stress 个分量很小，但晶格 a/b/c 漂得严重
    • bands NSCF 的 VBM 跟 SCF 的 fermi 差超过 0.1 eV（说明 NSCF k-path 上没覆盖真正的 VBM）
    • insulator 的 DOS 在 E_F 处不是 0
    • charge density 出现 NaN / spike（QE 在 stdout 通常给警告，但 LLM 经常漏看）
  • 有哪些是用 grep 几个关键词就能加进 checklist 的（最好实现）， 哪些必须 read+解析？
  • physics_sanity 现在覆盖了"已知 metal 被 fermi-slice 误判成 gap" + insulator/metal 倒置（Rules A/B）；📌 v2 update：05-20 又在 working tree 加了 Rule E（磁性矛盾）和 Rule F（vc_relax 残余应力） （详见 §4.2 v2 update 表）。这两条阈值——total_magnetization < 0.1 µB 判磁性塌缩、|P| > 5 kbar 判 vc_relax 残余——选得对吗？ 其他 metal/insulator 失配场景（Mott / Kondo / 半金属 / topological metal）要不要也写进规则？ 哪些只该 soft_warn 哪些必须 fail？

§6.3 · Step kinds 切分

请审 §4.3 那 7 种切片：

  • 现在的 7 种（scf / vc_relax / relax / bands / dos / collect / physics_sanity） 对应你日常的 QE 工作流是不是自然切片？
  • nscf 现在是嵌在 bands 和 dos 里的"内部子步"，要不要独立成一类？ （好处：dependency 更清晰；坏处：plan 变长，planner 更难写对依赖。）
  • bench 下半年要扩到 phonon (DFPT, ph.x) / 缺陷形成能 / 表面能 / NEB / HSE 杂化 / DFT+U 总能。每一类应该怎么切？例如：
    • phonon 是不是 scf → ph.x（DFPT 计算 dyn matrix）→ q2r → matdyn 四步？
    • 缺陷形成能要不要拆成 "perfect_scf + defect_scf + chempot" 三步？
  • "1 个 step 1 个 LLM session" 这个粒度——你觉得太细了还是太粗？ 粒度太细：planner 出错率高 + LLM 上下文切换成本；粒度太粗：单 step 容易超时 + retry 浪费大。

§6.4 · qe_input_lint 默认值 / 处理策略

请审 §4.4 那张默认值表：

  • vc-relax 三阈值（0.05 / 1e-4 / 1e-5）——是否过紧？ 会不会有 cell 接近 instability 的体系（如近相变温度的钙钛矿）反而该 用松阈值，否则 BFGS 在 noise 里震荡不收敛？
  • "只收紧不放松"（min(LLM_value, default)）这个策略—— 有没有合法物理场景是"题目就需要松收敛"的？我们想过的反例只有 "做一次粗扫探索 EOS 形状"，但 bench 题目目前没这种。
  • USPP ecutrho ≈ 10× 这个规则——是不是过于一刀切？ SSSP efficiency 库里有些 PAW 其实 8× 就够，有些 hard 元素（O / F） 可能需要 12×；要不要按元素查表更细？
  • occupations 我们当前是"仅警告不强改"——你觉得 advisor 高置信 （source=deterministic）时应该强改吗？什么样的"合法反例" 应该让 LLM 能 override 这个建议？
  • 有没有其他 hook 该加但我们没加的？例如：
    • 磁性体系自动注入 nspin=2, starting_magnetization 起点
    • 含 d/f 元素自动给个 U_value 候选值
    • k-mesh 太稀（小于 8×8×8 in 立方）自动加密
    • 大单胞自动调 mixing_beta / diagonalization='davidson'
  • 📌 v2 update：05-20 已经加了 Rule E（K_POINTS crystal_b 数据行 校验）（详见 §4.4 表）——专门接住 LLM 把 seekpath 输出粘错的常见情况。 这条规则逻辑没人审过，请你看：拒绝 spawn 然后回写诊断 vs 自动改写 vs 仅警告， 哪个动作更合理？另外，K_POINTS automatic / K_POINTS gamma 这两种是否也该加 sanity（automatic 的 6 个数 / gamma 的 0 个数）？

§6.5 · experience library 的 methodology-only 边界（v2 新增）

请审 §4.5 整节——这是 05-19 加的全新子系统，所有边界都是 shijie 一个人定的， 最需要 DFT 同行的眼睛。

  • validate_lesson 的 regex 边界：
    • 当前的 3 层过滤（元素符号 / required_keys / number+unit）能不能拦住"在 Si 上把 ecutwfc 调到 80 才让 gap 收敛"这类伪装成方法学的答案缓存？我们怕的是： LLM 学会用同义表达绕过 regex（比如"原子序数 14 的体系"代替 Si， 或者"高密度截断"代替 ecutwfc=80）。
    • QE keyword 白名单只覆盖 20 来个常用参数； Hubbard U / 磁性 / DFT+D 系列、ph.x 的输入参数都没加。要不要扩成两表 （核 + 扩展，扩展只在对应 task_kind 触发）？
    • "number+unit 一律视为答案"对不对？合法反例： "超过 0.5 GPa 的残余应力"这种"阈值描述"算方法学还是答案？ 当前会被 regex 误判成答案。
  • scope 4 维够不够：当前 {task_kind, crystal_system, n_atoms_bucket, step_kind}。物理上是否漏了：
    • spin / magnetism（有磁 vs 无磁，教训共享性差异巨大）
    • 元素组成（含 d/f / 含轻 / 含重 / 纯 sp）
    • 赝势家族（GBRV USPP vs SSSP-mixed vs PseudoDojo NC）
    • 电子相关强度（Hubbard U / hybrid / DMFT...）
    维度多 → 命中率低（多数 scope 上 0 条教训），维度少 → 教训误用。 你的 DFT 经验里，哪几个维度上同一条教训"不能跨"？
  • 教训长寿命问题：JSONL 永远 append、不过期。一条 2026 年写的"用 occupations='tetrahedra' 配 deltaE=0.01 eV"教训，半年后 QE 升级到 7.6（假设默认 smearing 算法变了）还成立吗？要不要：
    • 带 QE 版本戳，下次 load 时按版本过滤？
    • 每条教训带个"命中计数"，命中 N 次还没失效再升权重？
    • 就接受 stale 风险（反正 reflection 还能再写新教训覆盖旧的）？
  • 触发条件偏严：只在 verifier "先 fail 后 pass" 的 step 上触发反思。 一次性 pass 的"好运行"不写教训。你觉得：
    • 好运行也该提炼"关键配置回放"（哪些参数选对了）？
    • 还是这种过度提炼反而容易蜕变成答案缓存？
