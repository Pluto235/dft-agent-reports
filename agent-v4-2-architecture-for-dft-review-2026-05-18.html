1 · 起因：上一版 solver 老在哪些 DFT 细节上栽跟头

我们最早的版本（叫 v1）很简单：一个 LLM，给它三个工具 （bash 跑命令、write_file / read_file 读写文件）， 再丢一段几十行的"你是 QE 用户"的系统提示，让它读 bench 给的题目（task prompt）、 自己想、自己写 QE 输入、自己跑、自己判断收敛没收敛、自己交答案。形式上就是"边想 边干，干完看一眼，再想再干"。

v1 像你临时雇了一个远程同事，他通读过 QE 文档但没真做过项目。你把题目甩过去， 他会试着做；但你不在旁边，他怎么挑参数、查没查输出、什么时候放弃，全凭他自觉。

在 6 道最简单的题目（Si / NaCl / Cu 这类单元胞 vc-relax）上 v1 能拿到 0.7+ 的均分。 但稍微难一点，就会暴露这几类系统性问题——熟悉 QE 的同行一看就知道是怎么回事：

v1 上经常踩的坑对应的物理 / bench 后果
vc-relax 用了 QE 的默认收敛阈值 （press_conv_thr=0.5 kbar、forc_conv_thr=1e-3 Ry/Bohr、 etot_conv_thr=1e-4 Ry） cubic 系统的弛豫结构漂移 ≥0.01° 或 ≥0.05%，pymatgen 的 SpacegroupAnalyzer 把它降级为 trigonal/orthorhombic，空间群分扣 0.25。实测 ~25% miss 率。
SCF 不收敛 → 死循环 LLM 反复改 mixing_beta / 加 nstep，2 小时墙时间用完， 最后什么也没提交。
金属用了 occupations='fixed'， 或者绝缘体用了 smearing SCF 振荡 / 假收敛；smearing 把 gap 抹平，bench 评分崩。
T2 (bands+dos) 直接上来跑 bands 结构没先 vc-relax，bands 结果建立在错的几何上；或者 PBE+USPP 用了 ecutrho=4×ecutwfc（NC pseudo 的默认）→ 带隙漂 0.2–0.5 eV。
bash 命令里跑了 pip install / curl bench audit 要求纯离线运行，看见网络命令直接判 0 分。
LLM 自己说"我做完了" 漏交付字段 / 提交未收敛的 .out / 给的路径根本不存在 → bench 一查就 0 分。

把这几类坑列出来看会发现一个共性：不是 LLM 笨，而是 v1 这套架构里没有 "在它瞎来之前拦一下"的机制。LLM 写了松收敛的 vc-relax，没人事先收紧； 跑完它说"做完了"，没人独立查文件；它陷在死循环里，没人决定"该重试还是该换思路"。 v4.2 做的事，本质就是把这几个"没人做"的环节补上。

2 · v4.2 整体怎么工作的：三个 LLM 角色接力 + 两道前置功课

v4.2 不再是一个 LLM 单打独斗，而是分工流水线。bench 把一道题（比如 "算 Si 的 vc-relax 后晶格常数和空间群"）丢给 solver，solver 大致这样跑：

图 1 · v4.2 的整体流程。两道前置功课 + 三个 LLM 角色接力 + retry / replan 反馈环。  v4 另：每次 pw.x 跑完会自动解析结构化 facts（converged / energy / iter / force / stress / wall_time）写进 qe_facts.jsonl，verifier 直接读它，不再 grep .out 末尾。

三个 LLM 角色各管一段，互相不串台：

  • planner：只读题目，输出一份"分几步、每步做什么"的清单 （比如"第一步 vc-relax，第二步收集结果"）。它不读文件、不跑命令， 只决定"工作流是什么样的"。
  • executor：拿到 planner 写的某一步，去实际写 QE 输入、跑 pw.x、看输出。 每一步换一个全新的 LLM session——上一步算 vc-relax，下一步算 bands，executor 不会被上一步的细节污染。
  • verifier：用一份"这一步该长什么样"的清单 （比如 vc-relax 步骤必须有 bfgs converged in 字样、 cubic 系统 a/b/c 要在 0.01 Å 内一致）去查 executor 留下的文件。 通过就放行，不通过就给一句话原因（这句原因决定下一步怎么走）。

verifier 不通过怎么办？ 同一步先让 executor 看着 verifier 的原因重试， 最多重试 2 次。还不行就退到 planner，让它根据这个失败原因写一份新计划——这叫 "replan"， 最多 5 次。最终都救不回来才提交 incomplete。

v6 旁注 · verifier 的 default-pass 兜底（FIX-17）：verifier 在挣扎到 12 轮 budget 耗尽时，会做一次 fallback——如果 expected outputs 列表里的产物文件全部存在且每个 ≥ 1024 B，就 default-pass + 写 diagnosis 收尾， 而不是判 fail + 浪费一次 replan 配额。动机：bash hang 修好（v6 同期 FIX-15）后第一次暴露 verifier budget 耗尽案例，强 fail 会让 planner 误以为是步骤实质失败重写无意义计划。 这是一个 controversial fallback（宁错放也不浪费 budget），DFT 端的评审请见 §6.3。

v1 是"一个人闷头干"；v4.2 是 "项目经理（planner）写 spec， 工程师（executor）实施，QA（verifier）按清单验收，验收不过先返工再改方案"。 每个角色都可以独立换模型——目前 planner / executor 用 Claude Sonnet，verifier 用更便宜的 Haiku， 反思（见 §4.6）也用 Haiku。

整个过程跑完，agent 会写两个文件：

  • result.json 交给 bench 打分（bench 要的字段路径、文件内容）。
  • 全过程审计记录（每步谁说了什么、planner 写过哪几版计划、verifier 判过哪几次失败）。 我们用它复盘——任何一道题跑得好不好，都能精确还原。

3 · v4.2 怎么把上一版的"坑"一个个堵上

下面把 §1 那几类 v1 的坑，逐一对应到 v4.2 里是哪一个机制在挡。核心哲学： v1 把 DFT 知识"告诉" LLM 让它自己用；v4.2 把同样的知识写成代码层的强约束， LLM 想绕都绕不过去。

vc-relax 阈值太松，空间群退化 每次准备跑 pw.x 之前，agent 先把 LLM 写的输入文件交给 lint 检查； 发现 vc-relax 的三阈值不够紧，自动把它收紧到 press_conv_thr=0.05 / forc_conv_thr=1e-4 / etot_conv_thr=1e-5， 再 spawn pw.x。LLM 自己写多松都收紧。
SCF 死循环，跑墙时间 每步最多重试 2 次；耗尽就回去 planner 重写计划，最多 5 次。 全跪了就 submit incomplete，不让它拖死。
金属用 fixed / 绝缘体用 smearing 开题之前先做"金属还是绝缘体"的判别——化学式正则 + 元素列表 + 简单规则（含 d/f 元素 / 强电负性元素的组合）。结果作为 advisory 塞进 planner / executor 的提示词；executor 还会被自动监督 occupations 和这个建议是否一致。

4 · 我们写下来"代 agent 做主"的 6 类 DFT 决策

下面 5 个机制每一个都把具体的 DFT 物理判断写死到了代码或提示词里。 这就是这份报告想请你审的核心——这些默认值 / 判定规则，物理上对吗？漏了什么？

§4.1

写进 system prompt 的 QE 通用纪律

每个 executor 启动时拿到的提示词里，硬写了下面这些 "QE 老手都知道、文档第一页不写"对所有题目都适用的纪律。 （针对特定材料家族的额外说明书见 §4.2。） 新手 agent 第一次写 QE 输入很容易忘，所以我们直接告诉它"不准这样做、必须那样做"。

  • 赝势路径纪律：必须 pseudo_dir='./pseudos/'。 否则 pw.x 找不到赝势文件，Error in routine readpp。

§4.4

把 QE 工作流切成几个独立阶段

我们把 DFT 工作流切成了 7 个阶段（"step kind"），planner 写计划时 只能从这 7 个里挑。每个阶段对应它自己的 system prompt 加注、自己的工具白名单、自己的验收清单。

阶段做什么
scf自洽场——给定结构 → 电荷密度 + scf.out
vc_relax变胞弛豫——晶格 + 内坐标全松，三阈值强制收紧
relax固定胞、只松内坐标。很少用，多数情况 vc_relax 就够
bandsk-path NSCF + bands.x 后处理，依赖上游 SCF 的电荷密度
dos密集 k-mesh NSCF + dos.x，依赖上游 SCF
collect纯文件操作（cp / 改名 / 合并数据）；禁止调 pw.x / bands.x / dos.x
physics_sanityT2 任务自动加在最后，做物理一致性 cross-check（见 §4.3）

切分的好处是每个阶段可以独立写自己的 prompt + 验收清单——SCF 那段 prompt 只讲电荷密度，bands 那段只讲 k-path 和 nbnd，互不污染；新加阶段就加一个模块。

但 bench 今年要扩到 phonon (DFPT) / 缺陷形成能 / 表面能 / NEB / HSE 杂化 / DFT+U 总能这些；当前 7 种是否够、要不要新增、粒度对不对（nscf 当前是嵌在 bands / dos 里的内部子步，要不要独立成一类？）——见 §6.4。

跨阶段 invariant v6 新增

阶段之间是相互独立的（一阶段一 LLM session 不污染），但 v4.2 仍然维护 3 个跨阶段约束—— vc_relax → SCF → bands → DOS 这条链上的数据必须保持自洽。这 3 个 invariant 不属于任何单一 step_kind，是 advisor 在外部强制注入的。

  1. seekpath primitive cell 是 single source FIX-11： vc_relax 后的输出 cell 经 seekpath.get_path 转换得到 primitive； 后面 SCF / bands / DOS 全部基于这个 primitive 重算。 动机：AlAs X-CBM 漏读案例——vc_relax 后 cell 不是 seekpath primitive， K_POINTS crystal_b 写的高对称点（X / Γ）跟实际采样错位，bands.dat 里采不到真的 X。
  2. K_POINTS crystal_b basis 禁 transform FIX-11： bands.x 输入 K_POINTS crystal_b 必须写 seekpath 给出的 point_coords 原值；agent 不允许做任何 basis transform （即使 LLM 觉得"我换个 basis 表达 X 更整齐"）。 动机：path 必须跟 primitive cell 表达一致；transform 后即使数学等价，pw.x 内部 kgrid 索引会错位。

§4.6

让 agent 自动累积"踩坑笔记"

前 4 条机制都是我们写死的 DFT 知识。 这一条不同——它让 agent 把"先失败、后修好"的经历当成一次教训攒下来， 下次遇到类似情况自动复用。但有个硬约束：教训只能是"方法学" （比如"vc-relax 早期力震荡时调小 mixing_beta"），不能是答案 （比如"Si 的 ecutwfc 用 80 Ry"）。

图 2 · 自动累积笔记的闭环。失败 → 修复 → 反思 LLM 提炼一条方法学教训 → 存进对应阶段的笔记本 → 下次同类问题直接塞进开场记忆。

为什么硬要"方法学 only"

我们最怕的不是 agent 学不会，而是它学会的是答案。 比如有人无意中存了一条"Si 体系 ecutwfc 用 80 Ry 才能让 gap 收敛到 0.012 Ha"， 下次再跑 Si 直接命中——看起来"学会了"，本质是答案缓存。bench 评的是 agent 的通用 DFT 能力 而不是查表能力。

所以在写进笔记本之前，每条候选教训过三层正则：

  1. 先把合法的"方法学陈述"屏蔽。比如 mixing_beta=0.3 这种 QE 参数值是方法学（你做 DFT 的人就要知道用什么参数），先 mask 掉再判断。
  2. 剩下文本里不许出现任何元素符号。118 个全集——"Si" / "Fe" / "O" 都拒。 禁止把任何物理"答案 trigger"绑到具体材料上。
  3. 不许出现 "数字 + 单位" 形状（eV / GPa / Å / Ry / bohr / kbar）。 一条教训如果说"gap = 1.2 eV"那就是答案缓存。

这三层一起，大概率拦住常见的 答案泄漏。但 LLM 可能用同义表达绕开——这是 §6.6 想请你看的。

每条教训贴什么标签（用于检索）

标签取值为什么这样切

什么时候触发反思

只在"先 fail 后 succeed"的阶段上触发——也就是 verifier 之前已经判失败、 后来某次重试通过了。一次性通过的"好运行"不写笔记（也许那是运气，提炼出来可能误导）。 触发后给反思 LLM 三样东西：

  • 失败那次 verifier 的"原因"（比如"BFGS 前 3 步力震荡"）

  • 系统提示词里反复强调的硬约束（不准出现元素符号 / 不准带答案数值 / 一条教训 ≤ 200 字）

反思 LLM 如果觉得"这次没什么可教的"（比如失败那次只是打错了文件名）， 就主动 decline——空笔记本好过有毒的笔记本。
