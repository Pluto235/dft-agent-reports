关键背景

同模型，但走了不同的路径

v4.3 跑 deepseek-v4-pro 时走的是 legacy 手写 ReAct 循环（裸 bash，模型自己写 QE 输入、自己跑、 自己 grep 输出）。v5.1 的 should_use_sdk_path() 把 deepseek-* 也路由到 SDK 路径 （结构化工具 write_qe_input/run_pw/read_qe_output/verify_run/submit + 卡死 ledger + SCF 子 agent + 提交前验收门）。 这是 v5.1 有意的 go-forward 配置。所以下面看到的回归，是这条新路径上的 bug，不是路由失误—— 该修的是 bug，不是把 deepseek 退回 legacy。

3 · 逐题失分点（T1 + T2）

T1 失分点

  • 共同难题 Fe_bcc：两版都没拿满。v4.3 只有 0.35——弛豫到一个畸变胞（c、三个角、点群、磁矩全错， 只有 a/b、空间群编号、晶系对）；v5.1 好一些（0.739），但 c 偏长、空间群 / 晶系判错、磁矩偏。 bcc 铁的磁性 + 立方对称对这两条路径都偏脆。
  • v5.1 的零星小丢分：Cr_bcc 丢绝对磁矩、Fe2O3 丢空间群 / 点群。属于 0.075–0.15 量级的单字段抖动，不成系统。

根因 ① IO　Cr_bcc：提交了稠密网格的 bands.dat（−0.836，全场最大单项）

T2 的能带字段（band_gap/gap_type/vbm_k/cbm_k，共 0.75 权重）由打分器去解析 bands.dat。 该解析器需要的是高对称 k 路径上的本征值（带标签的 Γ/H/N/P… 点），typically ~180–210 个 k 点。 而 v5.1 在本批 Cr_bcc run（2026-06-01T13-57-23Z）最终提交的 bands.dat 头是：

$ head -1 .../2026-06-01T13-57-23Z_M3_T2_Cr_bcc_bands_dos/workdir/bands.dat
 &plot nbnd=  32, nks=  2052 /          ← 2052 = 稠密自动网格，不是 k 路径

2052 个 k 点是 DOS 用的稠密网格，不是能带路径。解析器拿到它认不出高对称点， 4 个能带字段全部返回 null：

report.json extracted_fields:  band_gap_ev=null  gap_type=null  vbm_k_label=null  cbm_k_label=null

对照 v4.3 同题：band_gap=0 / gap_type="metal" / vbm="metal" / cbm="metal" 全对（金属，0.969）。 v5.1 其实把高对称能带也算出来过（calcs 里有正确的 k 路径 NSCF），但提交 / 落盘环节用错了文件—— 这是结构化工具路径下 "calc_id → 输出文件" 多一层间接带来的文件张冠李戴，不是没算对。
（核对说明：另有一个 2026-06-01T07-38-46Z 的 Cr_bcc run，其 bands.dat 为正确的 211 点， 但 run_meta 显示它属于另一批次 82f32c…、agent v4_3，不在本次对照内。本批 v5.1 的 Cr_bcc 就是 2052 点那个。）

根因 ② IO　MnO：bands.dat 只写了自旋上分量（−0.327）

MnO 是 AFM rocksalt，nspin=2。正常 bands.dat 每个 k 点应有 2 个自旋通道的本征值 （这里 nbnd=30 → 每 k 点应 ~6 行）。v5.1 落盘的文件每 k 点只有 3 行（仅自旋上）：

transcript 末段自检：
  "K-vector lines: 270, Band data lines: 813, Band lines per kpt: 3.0"
  "WARNING: only 8943 values, expected 16531"   ← 少了一半（自旋下丢失）

解析器因数据量对不上而失败，band_gap_ev 返回 null → 该字段 0 分（丢 0.40）。 对照 v4.3 同题提交 band_gap=0.8777 eV（GT ≈ 0.867，命中，得 0.40）。 根因：v5.1 SDK 路径的能带后处理 / 解析没有正确处理 nspin=2 的双自旋通道—— 这和 Cr 是同一类毛病：bands.dat 的生成 / 解析在自旋极化体系下不可靠。

根因 ③ 参考系　金属的 dos_e_fermi 读偏（小权重，但系统性）

dos_e_fermi 容差 ±0.1 eV、权重仅 0.05。但模式很清楚：

根因 ④ 编排　Fe2O3 T2：算完了却不提交 → 超时

这道不是算不动。从日志看，SCF 收敛后，bands NSCF（517s）与 dos NSCF（2857s）都 JOB DONE， bands.dat 和 dos.dat 都已生成，此时墙钟还剩 ~8 小时。然后 agent 卡在反复 write_qe_input/编辑 + Python 自检里，始终没调 verify_run → submit； ledger 的 S3/S6 卡死信号点亮了也没能把它救回提交。最后一根稻草是 SDK 侧的 Stream closed 报错， solver 在墙钟耗尽时只能返回 best-effort 的 incomplete payload：

solver_stderr:  WARNING solve: wall budget exhausted during initial run; returning best-effort payload
harness.log:    [harness] solver exit_code=-1 status=timeout wall=36000.0s

根因：完成判定（completion gate）有漏洞——"产物已齐、预算尚足"时没有强制走到 submit， 叠加 SDK 流中断未做重试 / 兜底提交。代价不止这 0.097 分，更是白烧了 ~10 小时机时。

5 · 共性 / 结构性问题（两版都中，与版本无关）

把 v5.1 的 bug 修完，T2 也只能回到 ~0.5——因为还有一层天花板是两版共有的。这部分其实比版本回归更值得长期重视。

结构性天花板

所有绝缘体的 gap_type + VBM/CBM k 标签恒为 null

看表 4/表 5：对 Fe2O3、MnO、MnF2 这些真有带隙的材料，两版的 gap_type、vbm_k_label、cbm_k_label 全是 null—— 每道题白丢 0.35 权重。能"得到"这三项的只有金属（Cr/Fe），因为答案就是字符串 "metal"，不需要定位 k 点。 也就是说：两条路径目前都不会把 VBM/CBM 映射回高对称 k 点标签——它们能算出带隙大小（band_gap_ev 常对）， 却交不出带隙在哪、什么类型。这把每道绝缘体 T2 的上限压到 ~0.65。

  • Ni_fcc 金属被两版一致误判成小带隙半导体：v4.3 报 0.178 eV indirect、v5.1 报 0.274 eV direct， GT 是 metal。Ni 的窄 d 带贴着 E_F，采样 / 展宽稍偏就会"看出"一个假隙。这是 bands_dos 工作流对 窄带金属的共性脆点（对照 Cr/Fe 这种宽带金属两版都对）。
  • dos_curve 永远部分分：F1 在 0.21–0.85 之间，从不满分。属于 DOS 曲线对齐 / 展宽 / 网格的系统性偏差，两版同病。

含义：即便 solver 这边全修好，T2 的分数仍被"k 标签能力缺失"和"窄带金属误判"压着。 这两条要么给 agent 配一个 bands.x → seekpath 的 VBM/CBM 标签映射 helper， 要么回头看 rubric 对 k 标签的判定是否过严——属于 benchmark / 能力层面的事，单靠版本迭代修不掉。

6 · 为什么是 v5.1 这条路踩这些坑

同一个 deepseek，v4.3 没踩、v5.1 踩了，差别在路径形态：

表 9 · 两条路径对 bands.dat / 提交的处理差异
环节v4.3 legacy（裸 bash）v5.1 SDK（结构化工具）
文件管理模型自己 cd、自己持有 bands.dat 路径，"算的就是交的"经 calc_id 注册 + 开窗读，"算的文件"与"交的文件"之间多一层间接 → 易张冠李戴（Cr）
nspin=2 能带模型手写后处理，碰巧覆盖双自旋结构化后处理未稳妥处理双通道 → 只落自旋上（MnO）
提交闭环循环简单，算完即 grep + submitverify_run / gate / ledger 多步，"产物齐但没人按提交键"的状态可发生（Fe2O3）

讽刺的是：v5.1 为提升可靠性加的机制（结构化工具、ledger、子 agent、验收门），在 T2 上没帮上忙，反而引入了新的失败面。 这不是说这套机制错——而是它的文件流与完成判定还没在"bands_dos 这种多步、多文件、自旋极化"的场景下打磨到位。

关于"丢了三条 lesson"——给个诚实的边界：v5.1 把 deepseek 路由到 SDK 路径后，模型拿到的是 executor.md，而 v4.3 legacy 的 system_prompt 里那套逐任务的 ecut / k 网格密度公式 + "够用即可"的墙钟纪律 （以及 v5.1 写在 system_prompt.py 里、却只喂 legacy 路径的 V5.1-LESSON 块）SDK 路径都读不到。 这条属实（已在源码 + 实际 transcript 双向核对：SDK 路径的 Cr run 用了 ecut=60、各向同性 12³，没有那套公式）。 但要说清楚因果：本次 T2 的大额失分是上面 §4 那几个 IO / 编排 bug，不是这套 lesson 缺失直接造成的 （Cr 是立方，各向异性 k 本来也不影响它）。lesson 缺失更像一个潜伏的结构性风险，主要体现在 T1/长任务的墙钟上—— 本批 v5.1 的 Fe_bcc T1 跑了 35253s、Fe2O3 T1 直接撞 36000s，正是这类风险的影子。所以这条要修，但别把它当成 T2 回归的主因。

7 · 修复清单（按优先级）

P0bands.dat 提交校验：交给打分器的必须是 k 路径文件，不是稠密网格。 在 verify_run / 提交门加一条硬检查：被解析的 bands.dat 的 nks 要等于本次能带 NSCF k 路径长度（~180–210）， 若等于稠密网格规模（>1000）则判 hard_fail 并重放。预期收回：Cr 类 ~+0.84/题。
P0nspin=2 能带后处理 / 解析覆盖双自旋。 确保自旋极化体系的 bands.dat 同时落自旋上 / 下；解析端按 2×nbnd 校验行数，不足即报错重算。 预期收回：MnO 类 ~+0.33/题。
P1完成门兜底：产物齐 + 预算足 → 强制 verify_run→submit。 当 bands.dat/dos.dat 已存在且墙钟仍有余量却长时间没提交时，由 gate 主动触发一次验收并提交； 对 SDK Stream closed 做重试 / best-effort 提交，别让算完的活儿烂在手里。收回 Fe2O3 那道 + 省 ~10h 机时。
P2dos_e_fermi 取数源固定为 SCF（或与 GT 同一约定的那次计算）。 不要从稠密 NSCF/DOS 读 Fermi。收回金属 dos_Ef，~+0.05/金属题。
P2把 ecut / k 网格密度 / 墙钟纪律补进 SDK executor.md。 让 deepseek 走 SDK 路径时也拿到这套参数 scaffolding（主要降 T1/长任务墙钟风险，见 §6）。
benchmark（跨版本）补 VBM/CBM k 标签能力。 给 agent 一个 bands.x → seekpath 的标签映射 helper，让它对绝缘体也能交出 gap_type/vbm_k/cbm_k； 并复核 rubric 对 k 标签匹配是否过严。这条修的是 §5 的结构性天花板，v4.3、v5.1 同时受益。

8 · 结论与下一步

结论：
① "v4.3 分高"成立，但仅在 T2，且 79% 来自 Cr_bcc + MnO 两道，根因是 v5.1 SDK 路径的 两个 bands.dat IO bug（稠密网格文件 / 只含自旋上），不是物理或模型能力。
② T1 上 v5.1 反而略胜（0.906 vs 0.892），说明新底盘的物理不弱。
③ 还有一层两版共有的结构性天花板：绝缘体的 k 标签 / gap_type 恒空、窄带金属（Ni）误判——这部分单靠 solver 修不掉。
④ 不建议退回 v4.3。建议按 §7 修掉 P0/P1（Cr、MnO、Fe2O3 三类，工作量不大），预期 v5.1 的 T2 即回到 ~0.48 与 v4.3 持平、整体反超，然后在 M3 同一 6 材料子集上重跑一批做验证。

下一步建议顺序：P0 两个 IO 修复 → 在 Cr_bcc / MnO 两题做最小复现验证 → P1 完成门 → 全 6 材料 ×（T1+T2）重跑一个新批次 → 与本文表 2 对账。
