§2 什么是 agent，为什么 DFT 计算需要它

"Agent"这个词在 AI 圈里被滥用得厉害，先把它说清楚。

💡 概念

Agent / LLM-as-agent

普通的聊天机器人是"问一句答一句"。Agent 是把大模型放进一个循环里： 让它能调用工具（比如执行 shell 命令、读写文件、查数据库）， 看到工具结果后再决定下一步动作，直到任务完成或主动放弃。

简单说：能动手做事的 LLM 就叫 agent。

在本项目里：solver 是一个 agent；它收到 DFT 任务描述，自己写 QE 输入文件、 自己执行 pw.x、自己读输出、自己交答案。整个过程没有人工介入。

为什么 DFT 计算特别需要 agent？三个原因：

  1. 试错型任务：跑一遍 QE 可能因为参数不对、收敛不上、对称性识别错而失败。 失败不是终点，是新信息。聊天机器人无法消化这种"失败信号"，agent 可以。
  2. 多步骤工作流：T1 任务（结构优化）≈ 写输入 → 跑 pw.x → 读结果 → 看是否收敛 → 收敛则交， 不收敛则改参数重跑。T2（能带 + 态密度）更长。手写 if-else 太死；让 LLM 当"决策中枢"更灵活。
  3. 领域知识密集：QE 的输入格式、k-mesh 怎么选、ecutwfc 怎么定、收敛阈值多少合适——这些是"会就会，不会就乱猜"的知识。 LLM 看过很多 QE 教程，能复述这些知识，agent 框架让它能把知识转成动作。
你不在的时候，让一个懂物理的远程同事帮你跑题：他能查文档、写代码、看报错、修 bug、最后把结果给你。 他不是事先写死一个 shell 脚本，而是边想边做。Agent 就是这种"远程同事"的程序化版本。

§3 通用前置：ReAct 循环 + 工具调用

所有 agent 框架的"主循环"长得都很像，叫 ReAct（Reason + Act 的缩写）。 先把这个循环看明白，后面三版的差异就好懂了。

💡 概念

ReAct 循环

每一"轮"由 LLM 做一次推理，输出"我想做 X"，然后 agent 框架真的去做 X，把结果回喂给 LLM，进入下一轮。

用一句口诀：想一步 → 做一步 → 看结果 → 再想，直到 LLM 说"我做完了"。

在本项目里：v1、v3 都是经典的 ReAct；v5 的 Claude 路径是 ReAct 加强版（加了卡死检测和子 agent 调度）。

图 1 · ReAct 循环（Reason → Act → Observe → 重复）

这个循环里有四个名词很重要：

💡 概念

System Prompt（系统提示）

给 LLM 看的"开场说明书"——告诉它你的角色是什么、能调用哪些工具、要遵守什么规则。 它在对话最开头，每一轮 LLM 都能看到。

在本项目里：v1 的系统提示 42 行（基本就是"你是 QE 专家"加工具列表），v3 写了 377 行， 塞满了 QE 领域知识。所以同样的模型在两个版本下行为差异巨大。

💡 概念

Tool / Tool Call / Tool Result

Tool（工具）= 你提前注册给 LLM 的一个函数，附带 JSON schema 描述参数。 LLM 不能直接执行任何东西；它只能输出一个 tool_call（"我想用 bash 跑这条命令"）。 你的 agent 框架收到 tool_call 后真的执行，把结果包装成 tool_result 回喂给 LLM。

整个 agent 跟外界打交道的唯一通道就是工具。注册了几个工具，agent 就有几种动作；没注册的事它干不了。

在本项目里：v1 和 v3 都只注册了 4 个工具（bash / write_file / read_file / submit）。 v5 的 Claude 路径注册了 7 个（多了 5 个领域专用 MCP 工具）。

§4 v1-stable 长啥样：最小可行 agent

v1 是这个项目的原点。它的设计哲学是"能跑就行，别上花活"。 它的整个架构可以一图概括：

图 2 · v1-stable 架构（LLM + 4 工具的最小 ReAct）

它的工具表面（tool surface，即 LLM 能调用的所有工具）就 4 个：

工具作用签名简化版
bash跑一条 shell 命令bash(cmd: str, timeout_s: int = 900) → {stdout, stderr, exit}
write_file创建或覆盖文件write_file(path: str, content: str) → {ok}
read_file读文件（可分行切片）read_file(path: str, offset=0, limit?) → {content}

这 4 个工具的真实定义在 v1-archive/agent/loop.py 里：

TOOLS = [

        "name": "bash",
        "description": (
            "Execute a bash command. Each call is a fresh subprocess. "
            "Bare `cd <dir>` updates the persistent cwd for subsequent calls; "

]

每个工具就是一个 JSON 对象，告诉 LLM "你可以叫我 bash，传 cmd 进来"。 Anthropic SDK / OpenAI 都识别这种结构化描述。

系统提示也短得离谱——总共 42 行，全文如下（截取核心 20 行）：

# /home/shijie/solvers/v1-archive/agent/prompts/system.txt · L1-19
You are a DFT expert solving a benchmark task with Quantum ESPRESSO.

Environment already set up for you:
  - QE GPU binaries on PATH (pw.x, bands.x, dos.x) — wrapped via podman+NVHPC
    Just invoke them normally; the wrapper handles container launch.
  - Pseudopotentials at $PSEUDO_DIR (also symlinked into ./pseudos/).
  - Python with seekpath, pymatgen available.
  - Working directory has been set to your sandbox; relative paths are fine.

Your tools:
  - bash(cmd, timeout_s=900): run a shell command, get stdout/stderr/exit.
  - write_file(path, content): write (or overwrite) a file, creating parents.
  - read_file(path, offset=0, limit=None): read a file, optionally line-slice.
  - submit(task_outputs, notes?): terminate and hand back paths.

Workflow:
  1. Read the task from ./task_prompt.txt
  2. Plan your calculation, write QE input files, run pw.x (and friends)
  3. Verify convergence and sanity-check outputs (e.g., `grep "JOB DONE."`)
  4. Call submit with task_outputs keys as instructed

看完这段就明白 v1 的味道——"你是专家，环境给你了，工具给你了，自己看着办"。 没有任何 QE 特定的参数建议、没有错误恢复指引、没有失败兜底。一切赌在大模型自己的训练数据里。

§5 代码走读：v1 的启动顺序

看 v1 的 loop.py 主循环代码（简化掉 transcript / 多 provider 适配的细节）：

# /home/shijie/solvers/v1-archive/agent/loop.py · L96-186（简化版）
def run_agent_loop(llm, system_prompt, task_prompt_path, workdir, out_path,
                   max_turns, max_output_tokens):
    task_prompt = Path(task_prompt_path).read_text()
    messages = [{"role": "user", "content": task_prompt}]
    cwd = workdir

    for turn in range(max_turns):
        # 1. 让 LLM 推理这一轮要干啥
        resp = llm.call(
            messages=messages, system=system_prompt,
            tools=TOOLS, max_output_tokens=max_output_tokens,
        )

        # 2. 把 assistant 的回复（含 tool_calls）写回对话历史
        messages.append({"role": "assistant", "content": resp.text,
                         "tool_calls": resp.tool_calls})

        # 3. 逐个执行 LLM 想调用的工具
        tool_results = []
        for tc in resp.tool_calls:
            if tc.name == "submit":
                write_result_json(out_path, dict(tc.input))
                return 0                              # 成功退出
            result, cwd = _dispatch_tool(tc, cwd, bash_log)  # 真跑工具
            tool_results.append({"role": "tool",
                                 "tool_call_id": tc.id,
                                 "content": json.dumps(result)})

        # 4. 把工具结果喂回 LLM，进入下一轮
        messages.extend(tool_results)

    # 5. 轮数耗尽还没 submit → 标记 incomplete 退出
    write_result_json(out_path, {"incomplete": True,
                                  "reason": "turn_budget_exceeded"})
    return 2

就是一个 for 循环。每一轮做 5 件事：

  1. 请求 LLM（带上完整对话历史 + 系统提示 + 工具列表）
  2. 把 LLM 的回答（可能包含若干 tool_call）追加到对话历史
  3. 逐个执行 tool_call（真跑！）
  4. 把工具结果以 role: "tool" 消息也追加到历史
  5. 进入下一轮——LLM 现在能看到完整的"我说过 → 我做过 → 结果如何"链条

退出条件三种：

  • 正常：LLM 调用了 submit 工具 → exit 0
  • 提前结束：LLM 既没 submit 也没调任何工具（"我不想干了"）→ exit 2
  • 预算耗尽：跑满 100 轮还没 submit → exit 2 标记 incomplete

所有可移植的 agent loop 都长这样。这一节看明白，后面 v3 和 v5 都是在这个骨架上叠东西。

§6 v1 会在哪些地方翻车

v1 跑 baseline（148 题）的实测里，确实能跑完大部分，但失分集中在几类：

翻车 1：vc-relax 收敛阈值忘改 → 空间群被降级

QE 默认收敛阈值很松，结构松弛完毕后晶格会偏离对称几角度。pymatgen 用 SpacegroupAnalyzer 对最终结构识别空间群时，松一点就把立方判成三方——评分系统看到空间群编号不对，直接扣 0.25 分。

v1 的应对：在系统提示里劝 LLM 用更严的阈值。 但实测有 ~25% 概率 LLM（特别是 haiku 这种快但糙的模型）写输入时把这个细节漏掉了。 劝不动。

翻车 2：SCF 不收敛卡死循环

SCF 算不出来，LLM 试着改 mixing_beta、改 k-mesh、改 smearing……试 10 次还不行， 但它不会"主动放弃 + 交个不完美答案"，而是继续 retry，直到 4 小时上限被 harness reap， 什么也没交上。

这是 v1 一个典型痛点：没有"budget-aware termination"机制——agent 不知道自己还有多少时间预算， 也不知道何时该认输。

💡 概念

Context Window（上下文窗口）

LLM 一次能"看"的对话历史是有上限的。Claude Opus 4.7 是 1M token，Sonnet 4.6 是 200K， DeepSeek-v4-pro 是 128K……token 是模型的"字符单位"，1K token ≈ 750 个英文单词或 400 个汉字。

每一轮 ReAct 都把新消息追加到历史，对话越长，下一次请求的输入越大、越慢、越贵。 而且一旦超过窗口大小，模型就看不到最早的内容了，可能"忘记"任务目标。

在本项目里：v1 跑 SCF 死循环时，对话历史会膨胀到十几万 token，单轮调用变贵 5-10 倍。 v5 用 卡死检测 来提前打断这种死循环。

翻车 3：bash 调用网络命令被 audit 判违规

评测 bench 有条硬规则："agent 在沙盒里不准联网"——禁 curl / wget / pip install / git clone 等。 但系统提示只是"告诉"LLM 不要这么干，并没有真的拦住。某些情况下 LLM 会调 pip install pymatgen 来装包（其实包早就预装好了），一旦命中 bash.log 里有 pip install → 整道题判 0 分。

这三类问题就是 v3 主要要解决的——下一节开始。

§7 v3 的核心思路：不扩工具，扩知识

v3 给自己定了一条规则：

工具表面不变——还是 bash / write_file / read_file / submit 这 4 个， LLM 看到的"我能做什么"跟 v1 一模一样。
变的是知识和护栏：把领域经验、收敛规则、安全限制以三种形式塞进框架： 更长的系统提示、可被 import 的 Python 帮助库、bash 调用前后的 hook。

这种设计哲学叫"scaffolding around the loop, not in it"—— 循环本身保持简单，把所有领域知识做成"脚手架"放在循环外。具体有 4 层脚手架：

§8 脚手架 1 · 377 行系统提示

v3 把系统提示从 42 行膨胀到 379 行。新增的部分基本都是 QE 领域知识。 对比一下 v1 vs v3 的口吻：

v1（42 行）v3（379 行）
"Use the QE binaries on PATH. Workflow: read task → run pw.x → submit." "⚠ CRITICAL: NO HALLUCINATION OF RESULTS. ALL numerical results MUST come from a real pw.x run YOU launch. The grader reads your QE .out files directly — bogus numbers hurt nothing but your score, and the audit will flag a too-fast solver as cheating (qe_wallclock_ok must hold: real QE wall / total wall ≥ 0.5%)..."

v3 的系统提示是用 Python 函数动态生成的——根据当前是 GPU 还是 CPU 后端， 渲染出不同的指引。这是 v1 没有的：

377 行内容大致涵盖：

  • "不许编造结果"（带评分系统的 audit 解释——agent 知道 audit 会查它）
  • pseudo_dir 必须用相对路径（解释为什么默认路径会 fallback 到容器里不存在的位置）
  • CPU vs GPU 的 mpirun 差异（CPU 模式 wrapper 自动 mpirun，自己别再加；GPU 必须显式加）
  • vc-relax 推荐 ecutwfc / ecutrho / k-mesh / 收敛阈值的具体数值
  • SCF + bands.x + dos.x 的完整流水线模板（写一遍给 LLM 抄）
  • 常见 crash 的恢复指南（"如果 pw.x 在 5 秒内退出且没 JOB DONE，是 podman init 抖动，重试一次就好"）
  • 显式 5 阶段工作流：Smoke → Plan → Execute → Verify → Submit

这种"把领域专家写过的所有 tips 全塞进 system prompt"的做法看上去笨， 但实测对 v3 这种"通用工具 + 领域 LLM"的组合效果很好—— LLM 调用 bash 前不会再忘记 pseudo_dir = './pseudos/'，因为系统提示里写得明明白白。

§9 脚手架 2 · qe_tools.py 帮助库

v3 的工具列表跟 v1 一样只有 4 个，但它做了一件巧妙的事—— 在 solver 仓库根目录放一个 qe_tools.py，里面预先实现了 QE 常用操作：

# /home/shijie/solvers/current/qe_tools.py · 摘要
from qe_tools import (
    write_pw_input,         # 生成 pw.x 输入
    parse_pw_output,        # 解析 pw.x 输出
    write_pp_input,         # 生成 pp.x 输入（投影计算）
    write_dos_input,        # 生成 dos.x 输入
    write_bands_input,      # 生成 bands.x 输入
    fit_eos,                # 状态方程拟合
    plot_band_structure,    # 画能带
    plot_dos_data,          # 画 DOS
    render_structure_png,   # 渲染晶体结构图
)

LLM 调用方法很 hack——通过 bash 工具执行一个内嵌 Python heredoc：

# LLM 通过 bash 工具实际发出的命令长这样：
python3 <<'EOF'
from qe_tools import write_pw_input
write_pw_input(
    filename="scf.in",
    calc="scf",
    atoms=ase_atoms,
    pseudo_map={"Si": "Si.upf"},
    ecutwfc=60,
    ecutrho=480,
    kpts=[8,8,8],
)
EOF

这样 LLM 就不用从头默写 Fortran namelist 语法（QE 输入是出了名的难写—— 注释符是 ! 不是 #，浮点数是 1.0d-4 不是 1.0e-4， 布尔值是 .true. 不是 True）。

这种 trick 本质上是用 Python 函数库当"隐藏工具"——工具表面还是 4 个， 但 LLM 拥有一个完整的领域工具箱，只要通过 bash 中转。 v5 把它升级成正经 MCP 工具（§14 见）。

§10 脚手架 3 · qe_input_lint 预处理钩子（重点）

这是 v3 最聪明的设计，也是最有教学价值的一节。

💡 概念

Hook（钩子）

"钩子"是工具调用前或后插一脚的代码—— LLM 让框架做某件事时，框架先调你写的钩子，钩子可以检查、修改、甚至取消这次操作。

常见两种：pre-tool hook（工具执行前——比如做安全检查、参数清洗）、 post-tool hook（工具执行后——比如截断超长输出、改写错误信息）。

在本项目里：v3 的 qe_input_lint 是经典 pre-tool hook——它截在 bash 工具运行前， 检测到 LLM 要跑 pw.x，就翻开输入文件、自动收紧收敛阈值。

问题回顾

v1 的痛点之一：LLM 写 vc-relax 输入时忘了写紧收敛阈值，松弛完毕空间群被降级，扣 0.25 分。 解决思路两种：

  • 消极方案：在系统提示里劝它写紧些。v1 就是这么干的——25% 漏写率。
  • 积极方案：bash 工具收到 pw.x 命令时，自动读取它要用的输入文件， 检测到 vc-relax 但阈值太松 → 直接改文件、把 diff 反馈给 LLM。这就是 v3 的 qe_input_lint。

逻辑步骤（伪代码）：

# BashTool.run() 内部，subprocess 启动前：
def pre_run_hook(cmd: str):
    if not _PW_X_RE.search(cmd):
        return                                 # 跟 pw.x 无关，放过
    input_file = extract_input_filename(cmd)    # 从 "pw.x -in xxx.in" 抠出 xxx.in
    content = input_file.read_text()
    namelists = parse_fortran_namelists(content)
    if namelists["control"].get("calculation") != "vc-relax":
        return                                 # 不是 vc-relax，放过

    patched = False
    for key, default in _V3_TIGHT_DEFAULTS.items():
        nl = _NAMELIST_FOR[key]
        cur = namelists[nl].get(key)
        if cur is None or float(cur) > default:   # 没写或太松
            namelists[nl][key] = default        # 强制改成 v3 默认
            patched = True

    if patched:
        input_file.write_text(serialize(namelists))
        log_to_tool_result("---- qe_input_lint patched ----", diff)

重要细节：

  • "取严不取松"：如果 LLM 写的阈值比默认更严，保留 LLM 的；只要更松，强制收紧。
  • diff 反馈给 LLM：tool_result 里追加一段 --- qe_input_lint patched ---， 让 LLM 知道"你写的输入被我改了，新内容是这样"。这样下一轮如果 LLM 还要改这个文件，它能看到当前真实状态。
  • 只针对 vc-relax：scf / nscf / bands 这些不需要收紧，照原样放过。

实测效果：v1→v3 在能跑通的 cubic 材料任务上，分数从 ~0.75 提到 ~1.0（空间群不再被降级）。

§11 脚手架 4 · bash_guard 安全护栏

💡 概念

Guardrail / Safety filter（护栏）

跟 hook 类似——在工具调用前过一层"安检"，拒绝违反规则的调用。 区别在于：hook 倾向"加工 / 改写"，guardrail 倾向"直接拒绝"。

典型实现：白名单（只允许 ALLOWED 的程序）+ 黑名单（明确禁止 DENIED 的）+ 上下文规则（写文件必须在工作目录内）。

在本项目里：bash_guard.py 用三层防护——程序白/黑名单、路径限制、Python 代码内容扫描—— 防止 LLM 误调用网络命令、装包命令、提权命令。

  • 路径检查：write_file 目标必须在 cwd 子树内（防 ../../etc/passwd）
  • 敏感文件保护：即使是只读，~/.ssh/ / ~/.aws/ / .env / id_rsa 也拒绝

  • 命令替换递归：$(...) / `...` 内的子命令也走一遍检查（防 white-bypass）

这一层很重要——v1 时代曾经出过 LLM 真的去 pip install pymatgen（明明已经装好了） 导致 audit 判违规的事故。bash_guard 之后，这种"无心之失"在源头就被拦住。

§12 附加 · Prompt Caching 3 个断点

💡 概念

Prompt Caching（提示缓存）

每次调 LLM 都要把整个对话历史 + 系统提示一起送上去。但系统提示和工具定义是不变的—— 每次都重新计算它的 token 既慢又贵。

Anthropic / OpenAI 都支持在 prompt 里标"cache_control"断点—— 告诉服务端"从这里开始之前的内容是稳定前缀，缓存它"。后续请求命中缓存的 token， 价格折扣 90% 左右，速度也快很多。

在本项目里：v3 启用了 3 个 cache 断点（system / tools / user 任务描述）， 对 12 题 fast 测试集实测缓存命中率 97.4%——意味着每轮 LLM 调用只有 ~2.6% 的输入是"新算"的。

具体做法：在请求里给某些 message 加 cache_control: {"type": "ephemeral"}。 例如 v3 用三层缓存：

# 概念伪代码（实际在 v3 的 llm 适配层）
messages = [
    {"role": "system", "content": "You are a QE expert...",
     "cache_control": {"type": "ephemeral"}},          # 缓存断点 ①：系统提示
    {"role": "user", "content": tool_defs_as_text,
     "cache_control": {"type": "ephemeral"}},          # ②：工具定义块
    {"role": "user", "content": task_prompt,
     "cache_control": {"type": "ephemeral"}},          # ③：任务描述
    # …后面动态变化的工具调用/回复，不缓存
]

三个断点的设计思路：

  • ① 系统提示：377 行——一次任务里不变，多轮都能命中
  • ② 工具定义：四个工具的 JSON schema——不变
  • ③ 任务描述：本次跑的具体题目——单题内不变

断点越多越好吗？不一定——Anthropic 最多支持 4 个断点， 且每个断点至少要 1024 token 才值得缓存（不够长缓存失效成本更高）。三断点是经验最优。

§14 v5 的两个新东西：MCP 工具 + 自动验证

v5 跟 v3 比有两个根本性的改变：

  1. 把通用 bash 拆成 7 个领域专用 MCP 工具—— 让 LLM 不再操心"shell 该怎么拼"，直接用 typed 接口表达意图
  2. 在 submit 前强制跑程序级 verifier—— 不再相信 LLM 自己说"我跑好了"，用 Python 重新检查一遍

外加几个支撑设施：子 agent 调度、卡死信号检测、经验回放、双路径兼容老模型。 后面三节细讲。

💡 概念

MCP（Model Context Protocol）

MCP 是 Anthropic 牵头的协议，让 agent 工具用统一格式描述自己—— 跟 USB 一样，插上就能用，工具开发者和 agent 开发者解耦。

跟 v3 那种 "我自己注册一个 JSON schema" 比，MCP 工具： (a) 支持 async，(b) 自带类型校验，(c) 跨 agent 复用更方便（一个 MCP server 可被多个 agent 引用）， (d) Claude Agent SDK 原生支持。

在本项目里：v5 用 claude_agent_sdk.create_sdk_mcp_server 注册了 7 个 MCP 工具， 全部 async def do_xxx(args, ctx) 结构。

§16 v5 三大新机制深挖

16.1 SCF 收敛子 agent

💡 概念

Sub-agent / Agent Delegation（子 agent / 任务委托）

主 agent 遇到某类特定子问题时，启一个"专精 agent"去处理。 子 agent 有自己的系统提示、自己的工具子集、自己的轮数预算——做完了把结果还给主 agent。

好处：(a) 主 agent 上下文不被这个子任务的"试错记录"塞满； (b) 子 agent 可以用更对症的提示和工具（比如只给它 run_pw / read_qe_output，不给 bash）； (c) 模块化，便于单独迭代和测试。

在本项目里：v5 有个 convergence_fixer 子 agent，专门处理 SCF 不收敛—— 主 agent 看到 run_pw 返回 SCF failed，调 Agent("convergence_fixer", calc_id=...)， 子 agent 按 6 条收敛 playbook（改 mixing_beta、改 smearing、改 k-mesh、改 ecutwfc...）逐条试。

16.2 ProgressLedger 卡死检测（7 个信号）

💡 概念

Stuck Detection（卡死检测）

Agent 跑久了会"卡住"——同一动作做 3 遍、同一错误反复出现、token 烧光还没进展。 靠 LLM 自己意识到很难（它每次都觉得"再试一下应该就行"）。 解决方法：程序级监控——记录每轮动作的 hash，检测重复模式、错误累计、超时进展， 触发后主动告诉 LLM"你卡住了，去做 X 或 Y"。

在本项目里：v5 的 ProgressLedger 实现了 7 类信号 S1-S7， 任一触发就给 LLM 注入 "[ledger] You appear stuck (signal=Sx ...)" 消息，列出建议动作。

7 个信号是什么？看代码里的定义：

信号触发条件典型卡死场景
S1最近 6 轮里同一 (tool, payload_hash) 出现 ≥ 3 次反复用同样参数调 run_pw
S2最近 8 轮里 ≥ 6 次工具错误各种参数都不对，错误堆积
S3过去 600s 内没有任何 verify_run 返 pass10 分钟没拿到任何成功验证
S4最近 10 轮里同一 bash 命令（去掉 cd 前缀）出现 ≥ 3 次同一 shell 命令反复 retry
S5LLM 文本里 "let me try again" / "still failing" 等"挫败语"≥ 3 次/最近 4 轮LLM 自己都觉得卡了
S6最近 15 轮里没注册任何新 calc光在读读写写，没真跑 QE
S7输入 token 用量 > 预算的 70% 且还没拿到 verify_ok预算快烧光了

触发后给 LLM 注入的提示长这样：

[ledger] You appear stuck (signal=S4: identical bash cmd ≥3 in last 10 turns).
Recent calc_ids: calc_v_001, calc_v_002
Choices:
  (a) delegate to convergence_fixer subagent for calc_v_002
  (b) submit with incomplete=true and notes describing what you tried
DO NOT repeat the same tool args.

关键设计：同一信号只触发一次（已经告诉过 LLM 就不再骚扰），且写入 workdir/stuck.log 供事后复盘。

16.3 Verifier Gate + Replay 机制

💡 概念

Verifier Gate（验证 gate）

LLM 调 submit 不再直接结束。Driver 接管，用 Python 重新检查所有提交的输出文件： SCF 是否真收敛？力是否真低于阈值？空间群是否真没降级？JOB DONE 标记是否真的在文件尾？ 任何一项 hard_fail，给 LLM 反馈"你这个 submit 不达标，原因是 X，请修复"， 重新进入循环最多 N 轮（默认 2）。

本质上是把"自评"换成"他评"——LLM 自己说"done" 不算数，程序说 done 才算。

在本项目里：v5 的 verify_vc_relax / verify_bands_dos 是真实 Python 函数， 检查清单包括 "JOB DONE 出现"、"final force < 1e-4 Ry/Bohr"、"lattice drift < 0.5%"、 "space group 跟初始一致" 等。

Verifier 的位置如下图：

图 5 · v5 verify gate：submit → 程序级检查 → pass 落地 / hard_fail 复活

code 摘要：

                help="If verifier hard_fails, replay the SDK loop up to N times")

这个机制的关键好处是消除 LLM 自欺欺人—— v3 时代有不少题是 LLM 输出 "I have successfully completed the calculation" + submit， 但实际文件不完整或没 JOB DONE。v5 这种"程序裁判"直接绝杀。

16.4（彩蛋）Experience 经验库

💡 概念

Experience Replay / RAG-for-agent

之前跑过的成功任务记录下来，下次遇到相似任务时，把当时的成功操作摘要喂进系统提示—— 让 agent "看着历史成功案例办事"，少走弯路。

实现常用 embedding 相似度检索（Sentence Transformers MiniLM-L6-v2 等小模型）+ 阈值过滤。

在本项目里：v5 的 retrieve_for_task 把当前 task_prompt 编码 → 跟历史成功案例的 prompt embedding 比对 → top-K 且分数 ≥ 0.7 的塞进 system prompt 的特殊 section。可用 --no-experience 关掉做 A/B 对比。

§18 术语表

Agent
能调用工具、看结果、自主决定下一步的 LLM 系统。详见 §2。
ReAct
"Reason + Act"的简写。最朴素的 agent 主循环——LLM 推理 → 调工具 → 看结果 → 再推理。详见 §3。
System Prompt
对话最开头给 LLM 看的"说明书"——角色、可用工具、规则。详见 §3。
Tool / Tool Call / Tool Result
注册给 LLM 的函数 / LLM 想调它发出的请求 / 框架真跑完返回的结果。详见 §3。
Hook
工具调用前 (pre) 或后 (post) 插一脚的代码。可改写参数 / 改写结果 / 拒绝执行。详见 §10。
Guardrail
护栏——比 hook 更倾向"拒绝"。白名单 + 黑名单 + 路径检查等。详见 §11。
Prompt Caching
把不变的前缀（系统提示、工具定义）缓存在服务端，后续命中折扣 90%。详见 §12。
MCP（Model Context Protocol）
工具的标准化接口协议，类比 USB——agent 和工具开发解耦。详见 §14。
Sub-agent
主 agent 调用的专精 agent，有自己的提示、工具子集、轮数。详见 §16.1。
Stuck Detection
程序级检测 agent 是否"卡住"——重复动作、错误累计、token 耗尽。详见 §16.2。
Verifier Gate
submit 前用 Python 重新验证输出，hard_fail 时 replay 让 LLM 修。详见 §16.3。
Experience Replay
把历史成功案例 embedding 化，新任务来了检索 top-K 喂进 system prompt。详见 §16.4。
Context Window
LLM 一次能"看"的最大 token 数。详见 §6。
