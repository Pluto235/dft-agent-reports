三次 Rb 跑都在 physics_sanity_1 步耗尽 replan 配额后退出，提取到的"能隙"读数惊人地稳定：

三次提取数字几乎一字不差——这本身就说明问题与 agent 的随机决策无关，是确定性的下游评估错误。

对比 GT：/home/shijie/projects/bench/tools/gt_selfrun/canonical_gt_t2/M1_T2_Rb_bands_dos.json 期望 band_gap_ev=0、gap_type=metal、dos_e_fermi=2.0011 eV。agent 实跑 vc-relax 收敛后的 Fermi 能 = 1.99 eV，DOS NSCF 后 Fermi = 2.0475 eV，物理上完全正确。所以问题**不在 QE 计算**。

2.3 根因：奇数 Zvalence 把 bands 切错位

GBRV USPP 的 Rb 包含半芯态：4s² 4p⁶ 5s¹ → Zvalence = 9。workdir/bands_1/bands.out 头部能直接看到 number of electrons = 9.00。

bands NSCF 用了 smearing='mv'，QE 不打印 highest_occupied/lowest_unoccupied 行，evaluator.py 走 electron-count 分支：

# /home/shijie/projects/bench/harness/lib/evaluator.py:492-534
def _find_vbm_cbm_via_electron_count(eigenvalues_per_k, nelec, lsda):
    ...
    # line 515  当前实现：只拒绝非整数 nelec
    if lsda or abs(nelec - round(nelec)) > 1e-3:
        return None
    # line 517  Python banker's rounding：round(4.5) = 4
    n_val = int(round(nelec / 2))    # ← 9.0 → 4.5 → 4
    ...
    valence_top.append(s[n_val - 1])  # 第 4 本征值 = 4p 半芯顶
    conduction_bot.append(s[n_val])   # 第 5 本征值 = 5s 金属带底
    ...
    # line 530  仅当 VBM ≥ CBM 才兜底为 None

        return None
    # 半芯远在金属带下方 → 兜底不触发 → 返回伪能隙 10.989 eV
BENCH 问题在评估器：非自旋极化 DFT 下，奇数电子数物理上不可能有整数带占据——必然是金属。_find_vbm_cbm_via_electron_count 缺一道对奇数 nelec 的拒绝，导致它把 4p 半芯顶当 VBM、5s 金属带底当 CBM。

2026-05-20 06:460.631通过但低分最近这一次，gap=0.0088 eV

3.2 关键：Ge 的 GT 标的是 metal

查 GT：/home/shijie/projects/bench/tools/gt_selfrun/canonical_gt_t2/M1_T2_Ge_bands_dos.json

ground_truth:
  band_gap_ev   = 0.0      (tolerance 0.1)
  gap_type      = "metal"
  vbm_k_label   = "metal"
  cbm_k_label   = "metal"
  dos_e_fermi   = 7.7839

convergence_evidence.scan_points:
  ecutwfc=30→80 Ry × 5 个 scan point，全部 band_gap_ev = 0.0、gap_type='metal'

这不是 GT 录错——PBE+GBRV USPP 给金刚石 Ge 确实算出近零能隙（PBE 对 Ge 的 self-interaction 误差使间接 gap 缩水到接近 0），所以 GT 标 metal 是 PBE 实算结果的诚实反映。物理上正确的"半导体 Ge"和 PBE 实算结果"准金属 Ge"在 bench 这套范式下是冲突的。

3.3 证据：最近一跑骑在 1 meV 阈值边

2026-05-20 06:46 的 physics_sanity_1/sanity.json：

{
  "verdict": "pass",
  "details": {
    "advice_is_metal": false,                ← v4.2 advisor：半导体
    "advice_source":  "deterministic",
    "extracted_gap_type":  "direct",         ← bench 提取：direct gap
    "extracted_band_gap_ev": 0.0088          ← 数值在 0.1 容差内
  }
}

看一下 bench 怎么决定要不要短路成 metal：

# /home/shijie/projects/bench/harness/lib/evaluator.py:705 / :776
gap_value = max(0.0, lumo - homo)
if gap_value < 1e-3:                  # ← 1 meV 阈值
    return _metal_t2_result(...)      # 返回 gap_type='metal' + labels='metal'

Ge 在 PBE+GBRV 下的实算 gap 在 0 ~ 10 meV 之间抖（依赖 k-mesh 细节）。2026-05-16 那次拿 0.956 的 agent 报了 gap = 0.0000 → 触发短路 → 全套 metal 标签命中；2026-05-20 这次报 0.0088 → 刚好高于 1e-3 → 标 'direct'，VBM/CBM 用 seekpath 真实标签 → 3 个标签字段全部失分，只有 band_gap 数值（在 0.1 容差内）和 dos_e_fermi 命中。

3.4 根因：advisor × physics_sanity × bench 三方拉扯

v4.2 端 config_advisor 走元素表把 Ge 判为半导体（物理上正确），physics_sanity 又把"agent 算出 metal"标记为与 advisor 矛盾的 fail（05-19 那次）。即使 agent 算对了 PBE 数值，标签字段还是要按 GT 给的 'metal' 来填——而 advisor + physics_sanity 联手禁止 agent 这么填。

SOLVER v4.2 这条规则在"PBE 准金属"上必然失分：advisor 给出的是物理事实（Ge 是半导体），但 bench 评估的是 PBE 实算事实（Ge 在 PBE 下 ≈ metal）。这两个事实在 Ge / α-Sn / InSb 这类 narrow-gap-becomes-metal-in-PBE 元素上不一致。

3.5 影响范围

PBE 准金属一族都会撞这个边界：Ge、α-Sn、InSb、HgTe、PbTe、Bi₂Se₃ 等窄带半导体在 PBE 下都接近零隙。当前标准档命中 Ge 一个。

3.6 推荐修复

FIX-3a bench 侧（1 行） · evaluator.py:705 / :776：把金属短路阈值从 1e-3 抬到 1e-2（10 meV）。 
-    if gap_value < 1e-3:
+    if gap_value < 1e-2:    # 10 meV：吸收 PBE 准金属（Ge / α-Sn 等）的数值噪声
         return _metal_t2_result(...)
理由：1 meV 比 PBE 自身的能带误差小一个数量级；10 meV 是更合理的"数值上的零"界限。Ge 报 0.0088 / 0.0045 / 0.0001 任何一种都会稳定短路为 metal。
FIX-3b v4.2 侧 · config_advisor 给 PBE 准金属族打 gap_type_uncertain=True 标记，physics_sanity 在该标记存在时跳过 metal ⇄ semiconductor 的矛盾判定。在 advisor 的元素表里加一个 pbe_near_metal 字段，列入 Ge / α-Sn / InSb / HgTe / PbTe 等。

FIX-5 v4.2 侧 · 在 planner（或 executor 调 BashTool 的中间层）给每个 step 注入最小 timeout_s 下限。粗算： 
min_timeout_s(step) = max(
    step_kind_default,
    base × n_atoms × (kmesh_density / 30)³ × (nbnd / 16)
)

# step_kind_default
  scf:         300s   (5 min)
  vc_relax:    900s   (15 min — bfgs 迭代)
  bands_nscf:  600s   (10 min — k-path 一般稀疏)
  dos_nscf:   1800s   (30 min — dense kmesh + tetrahedra)
  bands_pp:     60s
  dos_pp:       60s
